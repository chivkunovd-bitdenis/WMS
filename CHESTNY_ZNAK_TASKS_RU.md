# Честный Знак — бэклог реализации (нарезка на задачи под Cursor Composer)

> Это **постановка для исполнителя**. Контекст/обоснование — в
> [docs/CHESTNY_ZNAK_DESIGN_RU.md](CHESTNY_ZNAK_DESIGN_RU.md). Здесь — что именно строить, по одной
> задаче за раз. Каждая задача = один PR (вертикальный слайс по [AGENTS.md](../AGENTS.md)).

## Как этим пользоваться (для исполнителя)

1. Берёшь **одну** задачу `Tx.y`. Не начинаешь следующую, пока текущая не «зелёная».
2. Соблюдаешь слои: API → `backend/app/api`, логика → `backend/app/services`, модели →
   `backend/app/models`, доступ к БД → `backend/app/db`, Celery → `backend/app/tasks`.
3. Гейты перед PR: `cd backend && ruff check . && mypy . && pytest`; `cd frontend && npm run build
   && npm run test:e2e`.
4. **Definition of Done (для любой задачи):** миграция применяется и откатывается; ruff+mypy+pytest
   зелёные; e2e зелёные; UI — единый MUI (эталон `FfProductsCatalogScreen.tsx`); добавлены тесты из
   раздела «Тест» задачи.
5. Не нарушаешь **инварианты** (ниже). Если задача будто бы требует нарушить инвариант — стоп,
   задача описана неверно.

## Инварианты (нельзя нарушать никогда)

- **И1.** Пул = одна ЧЗ-карточка = один GTIN. Код принадлежит ровно одному пулу (`pool_id`).
- **И2.** Связь «пул ↔ товары» — только ручная (`marking_pool_products`, M2M). Никогда не угадывать
  товар по GTIN/названию автоматически.
- **И3.** Остаток считается **на пул** (общий на все товары пула), не на товар.
- **И4.** Один КМ расходуется один раз. Печать N копий одного кода (на товар + на пакет) — это
  параметр печати, остаток уменьшается на 1 за единицу товара, не за копию.
- **И5.** Всё мультитенантно: каждый запрос/выборка фильтруется по `tenant_id`; для селлера — ещё
  по его `seller_id`.
- **И6.** Любое изменение статуса кода пишет запись в `marking_code_events` (ничего «молча»).
- **И7.** Личный ключ ЭЦП селлера на сервере не храним (см. подпись — §8.2 дизайна).

## Соглашения

- **API-префикс блока:** `/operations/marking-codes` (как сейчас).
- **Роли:** `fulfillment_admin`, `fulfillment_staff` (+право `shift_lead`), `fulfillment_seller`.
  Селлер видит только свой `seller_id` (через `get_effective_seller_id`).
- **Ошибки:** `HTTPException(status, detail="<machine_code>")`; фронт переводит `detail` в текст.
- **Статусы кода (`marking_codes.status`):** `available | reserved | printed | applied |
  introduced | shipped | transferred | defective | replaced | void`.
- **Типы событий (`marking_code_events.event_type`):** `imported | printed | reprinted | applied |
  introduced | shipped | transferred | defective | replaced | voided`.
- **Номер документа:** `ПРЕФИКС-ГГ-ММ-ДД-N` (напр. `УПАК-26-06-26-1`). Префиксы: `ПРИЕМ`, `ОТГР`,
  `УПАК`, `ЗАГРКМ`, `ПЕРЕМАРК`.

# ФАЗА 0 — Фундамент (разгрести «кашу»)

### T0.1 — Нумерация документов

**Зависит:** —
**Цель:** человеко-номер `ПРЕФИКС-ГГ-ММ-ДД-N`, стабильно хранится на бэке.
**Миграция:** таблица `document_sequences (id, tenant_id, doc_type str, date date, counter int,
unique(tenant_id, doc_type, date))`. Колонка `document_number str null` на `packaging_tasks`,
`inbound_intake_requests`, `marketplace_unload_requests`.
**Сервис:** `app/services/document_number_service.py` → `next_document_number(session, tenant_id,
doc_type) -> str`. Логика: атомарно `INSERT ... ON CONFLICT (tenant_id,doc_type,date) DO UPDATE SET
counter=counter+1 RETURNING counter`; формат `f"{PREFIX[doc_type]}-{yy}-{mm}-{dd}-{counter}"`.
**Интеграция:** при создании упаковки/приёмки/отгрузки — проставить `document_number`, если пуст.
**API:** отдавать `document_number` в существующих ответах этих документов.
**Фронт:** показывать `document_number` в заголовках/списках этих документов (где сейчас id/UUID).
**Acceptance:** Дано пустой счётчик. Когда создаю 2 упаковки в один день. Тогда номера
`УПАК-…-1` и `УПАК-…-2`; на следующий день снова `-1`.
**Тест:** pytest: 2 вызова подряд дают `-1`,`-2`; разные `doc_type` независимы; конкурентность —
без дублей. e2e: на экране упаковки виден `УПАК-…`.

### T0.3 — Загрузка кодов в пулы (рефактор импорта, без предвыбора товара)

**Зависит:** T0.1, T0.2, T0.5
**Цель:** убрать обязательный `product_id`; раскладывать файл по пулам (GTIN); писать событие
`imported`; присвоить документ `ЗАГРКМ-…`. (И1, И2)
**Сервис (`marking_code_service.py`):** `import_marking_codes(session, tenant_id, seller_id, *,
files: list[(filename, bytes)], pool_specs: list[{gtin, title, product_ids[]}], uploaded_by)`.
Логика: парс → нормализация → дедуп по `(tenant, cis)` → группировка по GTIN → `get_or_create`
пул(ы) → проставить `title`, привязать `product_ids` → создать коды со `status=available`,
`pool_id` → событие `imported` на каждый код (или агрегатно) → `MarkingCodeImport` (батч) с
`document_number=ЗАГРКМ-…`. Вернуть `{по каждому пулу: accepted, duplicates, invalid}`.
**API:** `POST /operations/marking-codes/import` (multipart): `files[]`, `pools_json` (массив
`{gtin?, title, product_ids[]}`), `seller_id?` (ФФ). **product_id больше не принимаем.**
**Фронт:** см. Э2 (T0.8).
**Acceptance:** Дано файл с 1000 кодов одного GTIN. Когда гружу, указав title и 3 товара. Тогда
создан 1 пул, 1000 кодов `available`, пул привязан к 3 товарам, событие `imported`, документ
`ЗАГРКМ-…`. Повторная загрузка тех же кодов → все в `duplicates`, 0 новых.
**Тест:** pytest: один GTIN; файл с двумя GTIN → два пула; дубликаты; неформат; привязка товаров.

шаблон»; **предпросмотр** 2–3 единиц ленты; кол-во; «Сохранить как шаблон» (T1.1).
**Кнопки:**

| Кнопка | Действие | API |
|---|---|---|
| Печать | Резерв+печать, отправить ленту на принтер (T1.6) | `POST …/packaging-lines/{id}/print` |
| Сохранить шаблон | Создать шаблон | `POST …/print-templates` |
| Запросить у селлера | Уведомление о нехватке (фаза 3; пока тост) | — |
| Отмена | Закрыть | — |

**Состояния:** уже печатали → кнопка «Перепечатать» (через T2.x, до фазы 2 — скрыта/заблокирована);
пул пуст → «Печать» disabled, виден только «Запросить».
**Acceptance:** Дано пул 32, нужно 50. Когда открываю печать. Тогда красный баннер «не хватает
18»; включаю «печатать доступные 32» → печатается 32, событие, остаток 0.
**Тест:** e2e: баннер нехватки виден; пресет «Парами» даёт предпросмотр пары одинаковых кодов;
печать уменьшает «Напечатано».

### T1.4 — Скан-печать (товар → код печатается сам)

**Зависит:** T1.2
**Цель:** ноль кликов: скан штрихкода товара → следующий код из пула печатается по шаблону по
умолчанию.
**API:** `POST …/scan-print` body `{packaging_task_id, product_barcode}` → резолв товар→строка→пул,
печать 1 единицы по дефолт-шаблону, вернуть `{codes, layout}` или `{shortage}`.
**Фронт:** в экране упаковки — поле фокуса «Сканируйте товар»; на скан → запрос → печать (T1.6);
счётчик «осталось по строке» −1; звук/цвет успех/ошибка.
**Acceptance:** Дано пул с кодами. Когда сканирую штрихкод товара. Тогда печатается 1 пара,
остаток −1, без диалогов.
**Тест:** e2e: эмуляция ввода штрихкода в поле → «Напечатано» +1.

### T1.5 — «Печать всех ЧЗ» по задаче упаковки

**Зависит:** T1.2
**Цель:** одна лента по всем строкам задачи в порядке сборки.
**API:** `POST …/packaging-tasks/{id}/print-all` body `{layout_json?}` → по всем строкам, требующим
ЧЗ, собрать коды, вернуть единый `{codes[], layout}`; нехватка по строке → в ответе перечень.
**Фронт:** кнопка «Печать всех ЧЗ» на задаче; перед печатью — сводка «будет N кодов, нехватка по 2
строкам».
**Acceptance:** Дано задача с 3 строками. Когда жму «Печать всех ЧЗ». Тогда одна лента по всем
строкам, события по каждой.
**Тест:** pytest: агрегирование по строкам; e2e: кнопка печатает и обновляет статусы строк.

### T1.6 — Рендер ленты по `layout` (`printMarkingCodeLabel.ts`)

**Зависит:** —
**Цель:** печать по раскладке `layout_json`: на каждую единицу — последовательность блоков
(`label`×copies, `cz`×copies), коды берутся по порядку, внутри единицы код одинаковый.
**Фронт:** расширить `buildMarkingLabelsDocument` принимать `layout`; блок `label` рендерит обычную
этикетку товара, `cz` — Data Matrix; профиль страницы 58×40.
**Acceptance:** для `layout {units:[{cz,2},{label,1}]}` и 2 единиц лента = `cz1,cz1,label, cz2,cz2,label`.
**Тест:** unit-тест экспандера последовательности; e2e: предпросмотр печати содержит ожидаемое число этикеток.

# ФАЗА 4 — Интеграция ЧЗ/МП (усилители §8.1 дизайна)

> Все задачи фазы 4 требуют сверки с актуальной документацией ГИС МТ/СУЗ/WB и реальных кредов.
> Подпись — оба способа (`ff_kep_mchd` и `manual`), см. §8.2 дизайна.

### T4.1 — Хранилище кредов селлера

**Зависит:** —
**Миграция:** `seller_marking_credentials` (поля из §5.9 дизайна: `cz_token_enc`,
`suz_oms_token_enc`, `marketplace`, `mp_api_key_enc`, `mchd_id`, `mchd_valid_until`,
`signing_method{ff_kep_mchd|seller_cloud|manual}`, `edo_route{edo_light_roaming_diadoc|diadoc_direct}`,
`auto_introduce bool`, `auto_emit_limit int null`). Шифрование at-rest (`app/core/crypto`).
**Фронт:** настройки в кабинете селлера: способ подписи, МЧД, маршрут ЭДО, флаги авто.
**Тест:** pytest: запись/чтение, значения не светятся в логах/ответах открытым текстом.

### T4.2 — Signing-service (manual + МЧД)

**Зависит:** T4.1
**Цель:** абстракция подписи: `sign(document, seller_credentials) -> signed`. Реализации:
`manual` (кладёт документ в очередь «на подпись» селлеру → он подписывает в кабинете),
`ff_kep_mchd` (подпись КЭП ФФ + МЧД через крипто-провайдера).
**API:** `GET …/signing/pending` (для селлера, manual), `POST …/signing/{id}/sign`.
**Acceptance:** при `signing_method=manual` документ ждёт подписи селлера; при `ff_kep_mchd`
подписывается без него.
**Тест:** pytest: ветвление по способу; manual создаёт pending.

### T4.3 — Авто ввод в оборот (ЧЗ API)

**Зависит:** T4.1, T4.2
**Цель:** по факту наклейки/отгрузки — батч-ввод в оборот через API ЧЗ; статус `introduced`.
**Сервис/Celery:** клиент ГИС МТ; собрать `applied`-коды по документу → отчёт о вводе в оборот →
подпись (T4.2) → отправка → на успех статус `introduced` + событие; на ошибку — уведомление.
**Acceptance:** Дано applied-коды отгрузки. Когда срабатывает авто-ввод. Тогда коды `introduced`,
события записаны; ошибка ЧЗ → уведомление ФФ/селлеру.
**Тест:** pytest с моком API ЧЗ: успех/ошибка.

### T4.4 — УПД ДОП → WB (ЭДО Лайт + роуминг Диадок)

**Зависит:** T4.3
**Цель:** сформировать УПД (ДОП, передача прав на КМ) по отгрузке, подписать, отправить;
**учесть, что WB не принимает напрямую из ЭДО Лайт — роуминг в Диадок**; трекать дедлайн **3 дня**
после приёмки поставки; статус `transferred`.
**Сервис:** собрать список КМ по поставке (точное соответствие код↔единица), сгенерировать УПД,
подписать, отправить по `edo_route`; хранить статус доставки/приёмки WB; таймер дедлайна →
уведомление при приближении.
**Acceptance:** Дано introduced-коды поставки. Когда формирую УПД и отправляю. Тогда статус
`transferred`, видна доставка в WB; дедлайн отслеживается.
**Тест:** pytest с моком ЭДО: генерация УПД, роуминг-маршрут, дедлайн-уведомление.

### T4.5 — Детектор «мёртвых» кодов

**Зависит:** T4.1
**Цель:** периодически тянуть статусы кодов из ЧЗ; помечать коды, что физически у нас, но в ЧЗ уже
выведены/заблокированы/использованы → не давать наклеить.
**Celery:** `marking_status_sync` по пулам; несоответствие → пометка `defective`/спец-флаг +
уведомление.
**Acceptance:** код, выведенный в ЧЗ, помечается и исключается из доступных; печать таким кодом
блокируется.
**Тест:** pytest с моком: «мёртвый» код исключён из выдачи.

### T4.6 — Авто-заказ кодов (СУЗ)

**Зависит:** T4.1, T3.3
**Цель:** по прогнозу (T3.3) и при `auto_emit` — заказывать эмиссию через СУЗ заранее, в пределах
`auto_emit_limit`; новые коды попадают в пул `available`.
**Acceptance:** при пробитии порога и включённом авто-заказе создаётся заказ эмиссии; полученные
коды появляются в пуле.
**Тест:** pytest с моком СУЗ: заказ в пределах лимита; превышение лимита → только уведомление.

### T4.7 — «ЧЗ под ключ»: статус сквозной цепочки

**Зависит:** T4.3, T4.4
**Цель:** в карточке поставки/отгрузки — цепочка: наклеено → введено в оборот → УПД отправлен →
принято WB. Зелёная, когда всё пройдено.
**Фронт:** компонент-степпер статусов на документе отгрузки + у селлера.
**Acceptance:** селлер видит сквозной статус маркировки поставки одной полосой.
**Тест:** e2e: степпер отражает статусы по мере прохождения.
