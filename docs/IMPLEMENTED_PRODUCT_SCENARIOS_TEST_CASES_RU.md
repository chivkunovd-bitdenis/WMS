# Тест-кейсы по реализованным сценариям продукта (только бизнес-уровень)

Документ разворачивает **[IMPLEMENTED_PRODUCT_SCENARIOS_RU.md](./IMPLEMENTED_PRODUCT_SCENARIOS_RU.md)** в набор проверок для **ручного тестирования и будущей автоматизации**. Здесь зафиксированы ожидаемое поведение, полномочия пользователя и ограничения **в той формулировке, в какой они даны в сценариях** — без спецификации API, кода и селекторов интерфейса.

**Как пользоваться документом**

- У каждого кейса стабильный **идентификатор** `TC-Sxx-yyy`: раздел S01 соответствует §1 сценариев, S02 — §2 и т.д.
- **Предусловия** — что должно быть до начала проверки (тенант, роль, данные).
- **Шаги** — действия пользователя в интерфейсе (формы, переходы, подтверждения).
- **Ожидаемый результат** — что видно на экране: списки, статусы, сообщения, недоступные действия.
- **Негатив и ограничения** — что пользователь **не должен** уметь сделать или какое сообщение об ошибке ожидается, если сценарий это предписывает.

**Роли** (как в документе сценариев): **администратор фулфилмента**, **продавец**. **Тенант** — организация фулфилмента; данные изолированы между тенантами.

Идентификаторы `TC-*` совпадают с англоязычным файлом **[IMPLEMENTED_PRODUCT_SCENARIOS_TEST_CASES_EN.md](./IMPLEMENTED_PRODUCT_SCENARIOS_TEST_CASES_EN.md)** — удобно вести общий каталог автотестов.

## S01 — Регистрация и первый вход

### TC-S01-001 Успешная первичная регистрация (будущий админ организации)

- **Роль:** новый пользователь без существующей учётной записи, который регистрирует организацию.
- **Предусловия:** нет (либо заранее подобраны уникальные email и slug).
- **Шаги:**
  1. Открыть публичный сценарий регистрации.
  2. Ввести название организации, URL-slug латиницей, email администратора, пароль.
  3. Подтвердить регистрацию основным действием формы (например «Зарегистрироваться» / аналог).
- **Ожидаемый результат:**
  - Создан тенант, пользователь считается авторизованным (сессия / токен с точки зрения сценария).
  - Открывается **дашборд** с данными профиля: как минимум email, название организации, роль.
  - Доступен переход к разделам **каталог** и **операции** (способ навигации — по сценарию S15).
- **Негатив и ограничения:** для позитивного сценария не задаются.

### TC-S01-002 Отказ в регистрации: занятые slug/email или неверный slug

- **Роль:** новый пользователь.
- **Предусловия:** slug и/или email уже заняты другим тенантом **или** slug не проходит правила продукта (слабый/некорректный).
- **Шаги:** повторить шаги TC-S01-001 с конфликтующими или некорректными данными.
- **Ожидаемый результат:**
  - Новый тенант под эту попытку не создаётся.
  - Пользователь видит **понятное** сообщение валидации или об ошибке, без «тихого» отказа.
- **Негатив и ограничения:** нет доступа к данным чужого тенанта.

## S02 — Вход и выход

### TC-S02-001 Успешный вход

- **Роль:** любой пользователь с действующей учётной записью.
- **Предусловия:** учётная запись создана.
- **Шаги:** открыть вход → ввести email и пароль → подтвердить вход.
- **Ожидаемый результат:** выдана сессия; после входа тот же уровень доступа, что и после регистрации (профиль, каталог и операции в рамках роли).

### TC-S02-002 Ошибка входа: неверный пароль

- **Роль:** пользователь, email которого известен системе.
- **Предусловия:** заведомо неверный пароль.
- **Шаги:** отправить форму входа с неверным паролем.
- **Ожидаемый результат:** отображается ошибка; в защищённую зону и к данным других тенантов доступа нет.

### TC-S02-003 Выход из системы

- **Роль:** авторизованный пользователь.
- **Предусловия:** активная сессия.
- **Шаги:** выполнить **выход** явным действием интерфейса.
- **Ожидаемый результат:** сессия на клиенте сбрасывается (как в сценарии); пользователь возвращается на **публичный** экран входа и регистрации.

## S03 — Склады и места хранения (ячейки)

### TC-S03-001 Админ создаёт склад

- **Роль:** администратор фулфилмента.
- **Предусловия:** выполнен вход под админом.
- **Шаги:** перейти к управлению складами (доступно только админу) → создать **склад** с названием и кодом.
- **Ожидаемый результат:** склад появляется в списке и доступен для выбора в дальнейших действиях.

### TC-S03-002 Админ создаёт ячейки (локации) внутри склада

- **Роль:** администратор фулфилмента.
- **Предусловия:** есть хотя бы один склад.
- **Шаги:** выбрать склад → создать одну или несколько **локаций** (коды ячеек).
- **Ожидаемый результат:** ячейки существуют как минимальные места учёта остатков и как цели привязки строк приёмки и отгрузки (по смыслу сценария).

### TC-S03-003 Продавец не ведёт склады в текущем интерфейсе

- **Роль:** продавец.
- **Предусловия:** вход под учётной записью продавца.
- **Шаги:** попытаться найти и выполнить те же действия по созданию склада и ячеек, что доступны админу.
- **Ожидаемый результат:** согласно сценарию, в текущем UI продавец **не** управляет складами — соответствующих экранов нет, либо действия недоступны (скрыты, неактивны, запрещены).

### TC-S03-004 Зависимость: без ячеек часть сценариев приёмки/отгрузки не проходит

- **Роль:** админ (и при необходимости продавец).
- **Предусловия:** склад есть, **ячейек нет** или их недостаточно для выбранного сценария.
- **Шаги:** попытаться пройти приёмку или отгрузку там, где сценарий требует предварительно завести ячейки.
- **Ожидаемый результат:** пользователь получает блокировку или подсказку до появления нужных ячеек; успешного завершения потока без ячеек быть не должно.

## S04 — Продавцы (B2B-клиенты)

### TC-S04-001 Админ создаёт запись продавца

- **Роль:** администратор фулфилмента.
- **Предусловия:** есть тенант.
- **Шаги:** создать запись **продавца** (имя).
- **Ожидаемый результат:** продавец есть для группировки, для будущих учётных записей продавца и для интеграции WB «на продавца».

### TC-S04-002 Учётная запись продавца не появляется автоматически вместе с записью продавца

- **Роль:** администратор.
- **Предусловия:** создана только запись продавца, без шага из раздела S12.
- **Шаги:** убедиться, что войти «как продавец» по этой записи без отдельного создания учётки нельзя.
- **Ожидаемый результат:** граница сценария — учётные записи продавцов заводятся **отдельно** (см. S12).

## S05 — Товары (SKU)

### TC-S05-001 Админ создаёт товар с обязательными полями

- **Роль:** администратор фулфилмента.
- **Предусловия:** вход админа; при необходимости заранее создан продавец для опциональной привязки.
- **Шаги:**
  1. Открыть форму создания товара (админская).
  2. Ввести наименование, код SKU, габариты (длина / ширина / высота в миллиметрах).
  3. При необходимости указать **продавца-владельца**.
  4. Сохранить товар.
- **Ожидаемый результат:** товар в **списке товаров**; при необходимости отображается или выводится **объём** (как в сценарии).

### TC-S05-002 После привязки к WB в списке видны идентификаторы WB (если связь есть)

- **Роль:** администратор.
- **Предусловия:** товар существует; выполнена привязка по сценарию S11 в рамках правил «тот же продавец».
- **Шаги:** открыть список товаров / строку товара.
- **Ожидаемый результат:** в списке могут отображаться **nmID WB** и **артикул продавца** (vendor code), как описано в сценариях.

### TC-S05-003 Продавец не пользуется админской формой «создать товар»

- **Роль:** продавец.
- **Предусловия:** вход под продавцом.
- **Шаги:** попытаться открыть и отправить админский сценарий создания товара.
- **Ожидаемый результат:** в текущем UI сценарий недоступен (нет доступа, форма скрыта или отправка невозможна).

### TC-S05-004 Продавец видит только разрешённые ему товары

- **Роль:** продавец.
- **Предусловия:** в каталоге есть товары разных продавцов и/или без привязки; учётка продавца привязана к одному продавцу.
- **Шаги:** открыть список товаров под продавцом.
- **Ожидаемый результат:** только товары своего продавца (и при необходимости — по правилам «без привязки» из сценария); чужих продавцов в рамках тенанта не видно.

## S06 — Входящая приёмка (inbound)

### TC-S06-001 Создание заявки на приёмку в статусе черновика: выбор склада

- **Роль:** админ или продавец — там, где интерфейс позволяет создавать заявку.
- **Предусловия:** склад существует; у пользователя есть право начать приёмку.
- **Шаги:** создать заявку на приёмку и выбрать **склад**.
- **Ожидаемый результат:** существует **черновик** заявки на приёмку.

### TC-S06-002 Добавление строки: товар и ожидаемое количество

- **Роль:** админ или продавец — в рамках полномочий.
- **Предусловия:** открыт черновик; товар есть в каталоге и доступен для выбора этой ролью.
- **Шаги:** добавить строку: товар, ожидаемое количество; **ячейку хранения** указать сразу или оставить на потом.
- **Ожидаемый результат:** строка отображается в заявке; товар и количество отображаются корректно.

### TC-S06-003 Запрет второй строки с тем же товаром в одной заявке

- **Роль:** админ или продавец.
- **Предусловия:** в черновике уже есть строка по товару P.
- **Шаги:** добавить ещё одну строку с тем же товаром P в **той же** заявке.
- **Ожидаемый результат:** операция не выполняется; пользователь видит явный отказ или сообщение об ошибке.

### TC-S06-004 Отправка заявки на приёмку на склад

- **Роль:** админ или продавец — где разрешено.
- **Предусловия:** черновик с корректными строками.
- **Шаги:** **отправить заявку** (передать на обработку складу).
- **Ожидаемый результат:** заявка переходит в состояние, пригодное для складской обработки (в терминах сценария — после «отправки»).

### TC-S06-005 Назначение или смена ячейки у строки (если статус позволяет)

- **Роль:** как правило админ для складских шагов; у продавца часть действий может быть только на просмотр.
- **Предусловия:** заявка в допустимом статусе; у строки ячейка может отсутствовать.
- **Шаги:** назначить или сменить **ячейку** у строки, когда интерфейс это разрешает.
- **Ожидаемый результат:** в строке отображается новая ячейка; при разрешённом статусе операция завершается успешно.

### TC-S06-006 Частичная приёмка количества по строке

- **Роль:** админ (у продавца — по ограничениям сценария).
- **Предусловия:** заявка в состоянии, когда допускается приём; выполнены требования сценария к строкам и ячейкам.
- **Шаги:** зафиксировать **приём** только **части** ожидаемого количества.
- **Ожидаемый результат:** накапливается принятое количество; частичная приёмка допускается.

### TC-S06-007 Проводка приёмки: остатки и движения

- **Роль:** админ (у продавца проводка может быть запрещена — см. сценарий).
- **Предусловия:** заявка готова к проводке по правилам продукта; назначения ячеек выполнены там, где требуется.
- **Шаги:** выполнить **проводку** (доведение непринятого остатка по правилам приложения).
- **Ожидаемый результат:**
  - **Остатки увеличиваются** в назначенных ячейках согласно правилам.
  - **Движения** видны в соответствующих журналах.
  - При необходимости проверить фрагмент **остатков по ячейкам** для затронутых локаций (вид админа).

### TC-S06-008 Ограничения продавца на проводку и часть складских действий

- **Роль:** продавец.
- **Предусловия:** есть заявка, которую админ мог бы провести.
- **Шаги:** попытаться выполнить **проводку** и другие критичные складские действия по приёмке.
- **Ожидаемый результат:** там, где UI ограничивает продавца, действия недоступны; исполнение остаётся за фулфилментом.

## S07 — Перемещение остатков между ячейками

### TC-S07-001 Успешное перемещение внутри одного склада

- **Роль:** администратор фулфилмента.
- **Предусловия:** в одном складе есть две ячейки; в ячейке-источнике достаточно **доступного** остатка.
- **Шаги:** открыть сценарий перемещения → выбрать ячейку **откуда**, **куда**, товар и количество → подтвердить.
- **Ожидаемый результат:** у источника остаток уменьшился, у назначения увеличился на то же число; есть движения типа **перемещение**; суммарный остаток по тенанту не меняется.

### TC-S07-002 Отказ: количество больше доступного (с учётом резерва под отгрузку)

- **Роль:** администратор.
- **Предусловия:** в ячейке есть фактический остаток, но **доступно** меньше из-за **резерва** под исходящую отгрузку (связка с S09).
- **Шаги:** запросить перемещение на количество **больше доступного**.
- **Ожидаемый результат:** операция не выполняется или ограничивается по правилам; пользователь информирован.

### TC-S07-003 Отказ: перемещение между разными складами в этом потоке

- **Роль:** администратор.
- **Предусловия:** ячейки «откуда» и «куда» относятся к **разным** складам.
- **Шаги:** попытаться выполнить перемещение в рамках описанного сценария.
- **Ожидаемый результат:** сценарий не поддерживается — операция невозможна.

### TC-S07-004 Продавец не выполняет перемещение между ячейками

- **Роль:** продавец.
- **Предусловия:** вход под продавцом.
- **Шаги:** попытаться найти и выполнить перемещение остатков.
- **Ожидаемый результат:** в текущем UI возможность отсутствует.

## S08 — Исходящая отгрузка (outbound)

### TC-S08-001 Черновик заявки на отгрузку: выбор склада

- **Роль:** админ или продавец — где разрешено создавать черновик.
- **Предусловия:** склад существует.
- **Шаги:** создать **заявку на отгрузку**, выбрать склад.
- **Ожидаемый результат:** есть **черновик**; при необходимости к заявке подтягивается **продавец** с первой строки (как в сценарии).

### TC-S08-002 Строка отгрузки: товар, количество, опционально ячейка отбора

- **Роль:** админ или продавец — в рамках полномочий.
- **Предусловия:** черновик открыт; товар выбирается; при назначении ячейки соблюдаются правила остатков.
- **Шаги:** добавить строку: товар, количество; при необходимости указать **ячейку отбора**.
- **Ожидаемый результат:** строка в заявке; когда ячейка задана в черновике или после отправки заявки, срабатывает **резервирование** против **доступного** остатка (подробнее S09).

### TC-S08-003 Запрет второй строки с тем же SKU в одной заявке

- **Предусловия:** в черновике уже есть строка по товару P.
- **Шаги:** добавить ещё одну строку с тем же P.
- **Ожидаемый результат:** отказ — не более одной строки на товар в заявке.

### TC-S08-004 Запрет смешения разных продавцов в одной заявке на отгрузку

- **Предусловия:** строки относятся к разным продавцам.
- **Шаги:** собрать такие строки в одной заявке.
- **Ожидаемый результат:** заявка не принимается системой (явный отказ).

### TC-S08-005 Отправка заявки на отгрузку: перепроверка резервов

- **Роль:** админ или продавец — где отправка разрешена.
- **Предусловия:** черновик с корректными строками; соблюдены правила остатков и резервов.
- **Шаги:** **отправить заявку** (переход в состояние после отправки на склад, в сценарии — submitted).
- **Ожидаемый результат:** заявка в состоянии «отправлена»; **резервы перепроверены**.

### TC-S08-006 Частичная отгрузка по строке (админ)

- **Роль:** администратор фулфилмента.
- **Предусловия:** заявка отправлена; строки допускают отгрузку.
- **Шаги:** выполнить **частичную отгрузку** по строке.
- **Ожидаемый результат:** остаток в ячейке уменьшается; отгруженное количество обновляется; **резерв** сокращается согласованно.

### TC-S08-007 Проводка остатка по неотгруженному (админ)

- **Роль:** администратор.
- **Предусловия:** у строк назначено хранение; по правилам остаётся неотгруженное количество.
- **Шаги:** выполнить **проводку остатка** (отгрузить всё ещё неотгруженное по строкам с назначенным хранением).
- **Ожидаемый результат:** неотгруженное по допустимым строкам отгружено; движения связаны со строками отгрузки.

### TC-S08-008 Полная отгрузка: заявка проведена и закрыта для дальнейшей отгрузки

- **Предусловия:** по правилам строк всё отгружено.
- **Ожидаемый результат:** заявка в состоянии **posted** (в терминологии сценария): дальнейшая отгрузка по ней не предполагается.

### TC-S08-009 Продавец не отгружает строки и не проводит заявку на отгрузку

- **Роль:** продавец.
- **Предусловия:** есть заявка, которую админ мог бы отгрузить или провести.
- **Шаги:** попытаться выполнить отгрузку строк или проводку.
- **Ожидаемый результат:** в текущем UI продавцу это **недоступно**.

## S09 — Резерв и «доступный» остаток

### TC-S09-001 Отображение фактического остатка, зарезервированного и доступного (где показывается)

- **Роль:** админ (и продавец — если балансы ему отображаются по правилам видимости).
- **Предусловия:** в ячейке есть остаток; при необходимости активны резервы под отгрузку.
- **Шаги:** открыть отображение балансов (глобально или в контексте, например подсказки после проводки приёмки).
- **Ожидаемый результат:** пользователь видит **фактический (on hand)**, **зарезервировано (reserved)** и **доступно (available)** — в той мере, в какой это отдаёт продукт.

### TC-S09-002 Вторая заявка на отгрузку не резервирует больше, чем доступно в ячейке

- **Предусловия:** заявка A зарезервировала большую часть остатка ячейки C; заявка B обращается к той же ячейке.
- **Шаги:** попытаться зарезервировать под B количество, превышающее **available** в C.
- **Ожидаемый результат:** явный отказ или блокировка; «перерезервировать» сверх доступного нельзя.

### TC-S09-003 Перемещение ограничено доступным, если отгрузка резервирует ячейку

- **Предусловия:** исходящая отгрузка резервирует остаток в ячейке C.
- **Шаги:** попытаться переместить из C количество **больше доступного**.
- **Ожидаемый результат:** согласовано с S07 — переместить больше **доступного** нельзя.

### TC-S09-004 Снятие резерва при отгрузке, удалении черновой строки или проводке/закрытии

- **Предусловия:** черновая строка отгрузки с резервом **или** отправленная заявка с частичной отгрузкой.
- **Шаги:** (а) отгрузить часть количества; (б) админ удаляет **черновую** строку (S10); (в) довести сценарий проводкой/закрытием до нулевого резерва.
- **Ожидаемый результат:** зарезервированное уменьшается или обнуляется по правилам сценария; остаток снова доступен для других заявок и перемещений.

## S10 — Удаление строки исходящей отгрузки (только черновик)

### TC-S10-001 Админ удаляет строку у черновика заявки на отгрузку

- **Роль:** администратор фулфилмента.
- **Предусловия:** заявка на отгрузку в статусе **черновик**; есть хотя бы одна строка.
- **Шаги:** удалить строку явным действием интерфейса.
- **Ожидаемый результат:** строка исчезает из списка; резерв по ней снят; если это была последняя строка — **привязка продавца к заявке может сброситься** (как в сценарии).

### TC-S10-002 Удаление строки недоступно не в черновике

- **Роль:** администратор.
- **Предусловия:** заявка в состоянии **после отправки** (submitted) или **проведена** (posted).
- **Шаги:** попытаться удалить строку тем же способом, что в черновике.
- **Ожидаемый результат:** действие недоступно; нужны обычные сценарии отгрузки и проводки.

## S11 — Импорт Wildberries (только чтение)

### TC-S11-001 Предпосылка: без записи продавца интеграция «на продавца» не задаётся

- **Роль:** администратор.
- **Предусловия:** сравнить ситуацию «нет ни одного продавца» и «продавец есть».
- **Шаги:** попытаться пройти сценарий выбора продавца для WB и сохранения токенов без созданного продавца (где это применимо).
- **Ожидаемый результат:** для интеграции нужен хотя бы один **продавец**; привязка интеграции — **на продавца**.

### TC-S11-002 Сохранение токенов: секреты в интерфейсе не показываются

- **Роль:** администратор.
- **Предусловия:** выбран продавец; токены получены вне приложения (тестовые или боевые — по среде).
- **Шаги:** вставить и **сохранить токены** через соответствующее действие UI.
- **Ожидаемый результат:** поведение соответствует сценарию: на сервере хранение зашифровано (это не проверяется глазами), в UI — **флаги** вроде «токен сохранён», **без** отображения секретов в открытом виде.

### TC-S11-003 Синхронизация карточек: фоновая задача и краткий итог

- **Шаги:** запустить **синхронизацию карточек** (в MVP — первая страница, как в сценарии).
- **Ожидаемый результат:** не «мгновенный синхронный ответ», а цикл: старт → ожидание завершения (опрос статуса) → состояние **успех или ошибка** → **краткий текстовый итог**.

### TC-S11-004 Синхронизация FBW-поставок: тот же паттерн, что у карточек

- **Шаги и ожидаемый результат:** аналогично TC-S11-003 для сценария синхронизации поставок.

### TC-S11-005 Просмотр списков импортированных карточек и поставок

- **Шаги:** после синхронизации открыть списки **импортированных карточек** и **импортированных поставок**.
- **Ожидаемый результат:** видны снимки данных в интерфейсе (как в сценарии — хранение в БД).

### TC-S11-006 Привязка внутреннего SKU к nm_id WB (один продавец с товаром)

- **Предусловия:** есть внутренний товар и импортированная карточка для **того же продавца**, что требуют правила привязки.
- **Шаги:** выполнить **привязку SKU** к **nm_id** (и правила артикула продавца — как реализовано в продукте).
- **Ожидаемый результат:** связь отражается в интерфейсе; в списке товаров могут появиться поля WB (см. TC-S05-002).

### TC-S11-007 Границы MVP: нет создания карточек и поставок в WB из приложения

- **Шаги:** поискать в интерфейсе действия **создать** карточку каталога WB или поставку WB из этого продукта.
- **Ожидаемый результат:** таких сценариев нет — только импорт и чтение; пагинация, расширенные поля, печать этикеток **вне** заявленного среза (проверка отсутствия или явной пометки «не в MVP» — по факту продукта).

## S12 — Учётная запись продавца и «кабинет»

### TC-S12-001 Админ создаёт учётную запись продавца, привязанную к записи продавца

- **Роль:** администратор.
- **Предусловия:** запись продавца создана.
- **Шаги:** создать **учётную запись продавца** (email и пароль) с привязкой к этому продавцу.
- **Ожидаемый результат:** можно войти под продавцом в контексте продавца.

### TC-S12-002 Вход продавца: дашборд в контексте продавца

- **Роль:** продавец.
- **Шаги:** войти с учётными данными продавца.
- **Ожидаемый результат:** дашборд отражает **контекст продавца**, а не полный админский вид по всему тенанту.

### TC-S12-003 Продавец видит только свои данные в списках

- **Шаги:** открыть товары, сводки по приёмке и отгрузке, движения, балансы.
- **Ожидаемый результат:** только данные **своего** продавца по правилам бэкенда; данные других продавцов того же тенанта недоступны.

### TC-S12-004 Черновики приёмки и отгрузки в рамках полномочий продавца

- **Шаги:** создать **черновики** приёмки и отгрузки по своему ассортименту там, где интерфейс разрешает.
- **Ожидаемый результат:** черновики создаются; **чувствительные шаги фулфилмента** (проводка, отгрузка, отправка заявки на отгрузку — по сводной таблице сценариев) у продавца остаются недоступны там, где сценарий это фиксирует.

## S13 — Глобальные движения и движения по заявке

### TC-S13-001 Список глобальных движений и обновление

- **Роль:** любой авторизованный пользователь в рамках своей видимости.
- **Шаги:** открыть **глобальные движения**; при наличии кнопки или жеста — **обновить** список.
- **Ожидаемый результат:** отображаются последние движения с типами вроде приёмки, перемещения, отгрузки (в пределах лимитов, оговорённых в сценарии).

### TC-S13-002 Движения по выбранной заявке на приёмку или отгрузку

- **Шаги:** выбрать конкретную заявку на приёмку или отгрузку → открыть связанный **список движений**.
- **Ожидаемый результат:** видны движения по этой заявке; прослеживаемость в рамках отображаемых ограничений.

### TC-S13-003 Сравнение видимости админа и продавца

- **Предусловия:** в тенанте есть движения и в масштабе организации, и в разрезе разных продавцов.
- **Шаги:** сравнить те же разделы под админом и под продавцом.
- **Ожидаемый результат:** продавец видит **свой** срез; администратор — **весь тенант**, как в сводной таблице сценариев.

## S14 — Фоновые задачи (общие и WB)

### TC-S14-001 Жизненный цикл задачи «сводка по движениям»

- **Роль:** администратор.
- **Шаги:** из доступного в UI места запустить **сводку по движениям (digest)** → дождаться завершения по отображаемому статусу.
- **Ожидаемый результат:** пользователь видит полноценный цикл (не имитацию мгновенного ответа): ожидание до **успеха или ошибки** и **краткий текст результата**.

### TC-S14-002 Жизненный цикл задач синхронизации WB

- **Ожидаемый результат:** тот же паттерн, что в TC-S14-001, для точек входа синхронизации WB (согласовано с S11).

### TC-S14-003 Примечание для проектирования тестов (среда выполнения)

- **Содержание:** при наличии Celery задачи могут идти на воркеров; без брокера возможно встроенное или локально-асинхронное выполнение. Автотестам разумно проверять **наблюдаемый пользователем жизненный цикл**, а не конкретную инфраструктуру, если среда не зафиксирована договорённостью.

## S15 — Навигация и вёрстка (UX)

### TC-S15-001 Навигация по разделам после входа

- **Роль:** авторизованный пользователь.
- **Шаги:** после входа воспользоваться **навигацией по разделам** (якоря) и перейти между **каталогом** и **операциями** на длинной одностраничной странице.
- **Ожидаемый результат:** переход к нужному разделу работает (фокус, прокрутка); «битых» якорей нет.

### TC-S15-002 Состояние загрузки профиля

- **Шаги:** открыть приложение в момент, когда ещё загружается профиль пользователя (в сценарии указана загрузка профиля, в том числе запрос `/auth/me`).
- **Ожидаемый результат:** показывается **индикатор загрузки**; формы входа и регистрации **не накладываются** друг на друга поверх контента.

### TC-S15-003 Дашборд админа ФФ — недельное планирование и «Поставки и загрузки» (Supply and Load)

- **Роль:** администратор фулфилмента.
- **Предусловия:** выполнен вход (тенант создан при регистрации или уже существует).
- **Шаги:**
  1. Оказаться на **дашборде** администратора фулфилмента (стартовый экран после авторизации в этой оболочке).
  2. Убедиться, что виден **недельный планирующий блок** и **краткие блоки** по приёмке и отгрузке (для нового тенанта списки могут быть пустыми).
  3. Перейти по пункту навигации **«Поставки и загрузки»** (единый список заявок на приёмку и отгрузку на склад МП).
  4. Нажать **«Создать выгрузку»** (продуктовый термин *download*: выгрузка ФФ на склад маркетплейса; пока заглушка).
- **Ожидаемый результат:**
  - Блок недели и сводки по документам **отображаются** и доступны для дальнейшей работы (например смена недели, если предусмотрено UI).
  - Экран списка открывается без ошибки; видна кнопка **«Создать расхождение»** (*diverge*); по заглушке показывается **информационное** сообщение (**ограничение:** документ в БД не создаётся).
  - Дальнейшие действия с уже существующими документами согласованы с разделом **операции** (см. S06/S08).
- **Негатив и ограничения:** кнопки новых типов документов без бэкенда; детальные правила приёмки/отгрузки — в S06/S08.

## Сводная матрица для дымового прогона по ролям

Используйте как чек-лист: каждая ячейка должна находить отражение хотя бы в одном явном кейсе выше.

| Область | Админ | Продавец | Продавцу недоступно (текущий UI) |
|---------|-------|----------|----------------------------------|
| Регистрация тенанта | Да | Н/П | Создание организации, если не предусмотрено |
| Склады и ячейки | Полное ведение | — | Создание складов и ячеек |
| Продавцы и учётные записи продавцов | Создание | — | Создание |
| Товары | Создание и просмотр по тенанту | Просмотр в своём срезе | Админская форма создания товара |
| Приёмка | Полный цикл там, где UI позволяет | Черновик, отправка, приём — по UI; проводка ограничена | Проводка и прочие шаги там, где запрещено |
| Отгрузка | Включая удаление строки черновика, отправку, отгрузку, проводку | Черновик и строки — где разрешено | Отправка заявки, отгрузка строк, проводка — где запрещено |
| Перемещение между ячейками | Да | — | Перемещение |
| WB: токены, синхронизация, привязка SKU | Да | — | Весь сценарий WB |
| Движения и балансы | Весь тенант | Только свой продавец | Данные чужих продавцов |
