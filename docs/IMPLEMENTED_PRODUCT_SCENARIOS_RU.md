# Реализованные сценарии продукта (бизнес-описание)

Документ описывает **что уже сделано** с точки зрения пользователя и процесса: роли, шаги, результат и ограничения. Это не спецификация API и не детали реализации.

**Роли**

- **Администратор фулфилмента** — оператор тенанта; полный доступ к каталогу и складским операциям; может пользоваться UI интеграции WB; может отправлять на обработку и проводить приёмку/отгрузку там, где продукт это допускает; может создавать учётные записи продавцов.
- **Продавец (seller)** — привязан к одной записи продавца; видит только данные в рамках этого продавца; может создавать черновики приёмки/отгрузки по своим товарам; не выполняет действия «только фулфилмент» (например проводка приёмки, отгрузка строк outbound), если UI это запрещает.

**Мультитенантность**

- Каждая регистрация создаёт **тенанта** (организацию фулфилмента). Все данные ниже изолированы по тенанту.

---

## 1. Регистрация и первый вход

**Актор:** новый администратор фулфилмента (без существующего аккаунта).

**Поток**

1. Пользователь вводит название организации, URL-slug (латиница), email админа, пароль.
2. Система создаёт тенанта и выполняет вход (токен сессии).
3. Пользователь попадает на **дашборд** (email, название организации, роль) и может перейти к разделам **каталог** и **операции**.

**Результат**

- Одна организация = один тенант; админ — первый пользователь.

**Ограничения**

- Уникальность slug и email; слабый или неверный slug отклоняется с понятным сообщением.

---

## 2. Вход и выход

**Актор:** любой пользователь с учётными данными.

**Поток**

1. Пользователь вводит email и пароль → проверка → выдаётся токен.
2. То же состояние после регистрации: профиль + разделы.
3. **Выход** очищает сессию на клиенте; пользователь снова на экране входа/регистрации.

**Результат**

- Повторные визиты: вход без повторной регистрации.

**Ограничения**

- Неверный пароль → ошибка; нет доступа к данным других тенантов.

---

## 3. Склады и места хранения (ячейки)

**Актор:** только администратор фулфилмента (в текущем UI продавец склады не ведёт).

**Поток**

1. Админ создаёт **склад** (название + код).
2. Админ выбирает склад и создаёт **локации** (коды ячеек) внутри этого склада.

**Результат**

- Ячейки — минимальные места учёта остатков и привязки строк приёмки/отгрузки.

**Ограничения**

- Без склада создание локаций не имеет смысла; без ячеек часть сценариев приёмки/отгрузки требует сначала их завести.

---

## 4. Продавцы (B2B-клиенты)

**Актор:** администратор фулфилмента.

**Поток**

1. Админ создаёт записи **продавцов** (имя).
2. Опционально: при создании **товара** админ может привязать его к продавцу (владелец).

**Результат**

- Продавцы нужны для коммерческой группировки, для **учёток продавцов** и для **интеграции WB** (на продавца).

**Ограничения**

- Учётные записи продавцов создаются отдельно (см. разд. 12).

---

## 5. Товары (SKU)

**Актор:** админ создаёт; продавец **видит** только товары своего продавца (или по правилам привязки в UI).

**Поток**

1. Админ вводит название, код SKU, габариты (Д/Ш/В в мм); опционально продавец.
2. Система сохраняет товар и показывает в списке (при необходимости — объём).
3. После привязки к WB (разд. 11) в списке могут отображаться **WB nmID / артикул продавца**.

**Результат**

- SKU — единица учёта остатков, резервов и движений.

**Ограничения**

- Пользователи в роли продавца не используют админскую форму «создать товар» в текущем UI.

---

## 6. Входящая приёмка (inbound)

**Актор:** админ и продавец — в рамках ограничений ниже.

**Типичный жизненный цикл**

1. **Создать заявку** — выбор склада → черновик заявки.
2. **Добавить строку** — товар, ожидаемое количество; **ячейку** можно указать сразу или позже.
3. **Отправить заявку** — переход в состояние, пригодное для обработки на складе.
4. **Назначить / сменить ячейку** у строки (если статус позволяет), если не задано при создании.
5. **Принять количество** по строке (допускается частичная приёмка) — фиксируется принятое количество.
6. **Проводка (post)** — по правилам приложения доводит остаток непринятого; **остатки растут** в назначенных ячейках; **движения** видны в журналах.

**Результат**

- Остаток появляется только после потоков, создающих положительные движения (приём/проводка — как реализовано).
- После проводки UI может показывать фрагмент **остатков по ячейкам** для затронутых локаций (вид админа).

**Ограничения**

- **Продавец** может быть только на чтение на проводке / части складских действий в зависимости от экрана (критичные шаги выполняет фулфилмент).
- Дублирующая строка с тем же товаром в одной заявке приёмки не допускается.

---

## 7. Перемещение остатков между ячейками

**Актор:** администратор фулфилмента.

**Поток**

1. Выбор ячейки **откуда**, **куда** (один склад), товар, количество.
2. Система переносит остаток: списание с источника, приход в назначение; пишет **движения типа transfer**.

**Результат**

- Физический остаток переезжает между ячейками без изменения суммарного остатка тенанта.

**Ограничения**

- Нельзя переместить больше **доступного** в исходной ячейке (фактический остаток минус **зарезервированное** под строки отгрузки — см. разд. 9).
- Нельзя переместить между разными складами в этом сценарии.

---

## 8. Исходящая отгрузка (outbound)

**Актор:** исполнение ведёт админ; продавец может создавать черновики и смотреть там, где ограничено.

**Жизненный цикл (концептуально)**

1. **Создать заявку на отгрузку** — склад; черновик (продавец может подтянуться с первой строки).
2. **Добавить строку** — товар, количество; опционально **ячейка отбора**.
3. Когда у строки задана **ячейка в черновике (или после отправки заявки)**, система **резервирует** количество (ожидаемое минус уже отгруженное) против **доступного** остатка в ячейке.
4. **Отправить заявку** — submitted; резервы перепроверяются.
5. **Частичная отгрузка** по строке (админ) — списание остатка, обновление отгруженного; резерв уменьшается соответственно.
6. **Проводка остатка** (админ) — отгружает всё ещё неотгруженное по строкам с назначенным хранением.
7. Когда всё отгружено, заявка переходит в **posted** (дальнейшая отгрузка по ней не предполагается).

**Результат**

- Остаток уменьшается при отгрузке/проводке; движения ссылаются на строки outbound.
- **Резерв** не даёт дважды «обещать» одни и те же свободные единицы двум заявкам в одной ячейке.

**Ограничения**

- Одна строка на товар в одной заявке outbound (без дублей SKU).
- Смешение разных продавцов в одной заявке отклоняется.
- **Продавец** в текущем UI не отгружает строки и не проводит (исполняет фулфилмент).

---

## 9. Резерв и «доступный» остаток

**Смысл**

- **Фактический остаток (on hand)** — физическое количество в ячейке.
- **Зарезервировано (reserved)** — количество, обещанное строкам отгрузки (черновик/submitted, ячейка назначена, ещё не полностью отгружено).
- **Доступно (available)** — фактический минус зарезервированное; используется для **перемещений** и для **добавления строк outbound**.

**Поведение для пользователя**

- API балансов (и подсказки в UI после проводки приёмки) могут показывать **reserved** и **available** рядом с фактическим.
- Если заявка A зарезервировала большую часть ячейки, заявка B не зарезервирует больше **available**.
- Если outbound блокирует остаток, **перемещение** из этой ячейки не может взять больше **available**.

**Снятие резерва**

- Отгрузка уменьшает зарезервированное; удаление **черновой** строки (админ) снимает резерв; проводка/закрытие строк убирает резерв по мере обнуления количеств.

---

## 10. Удаление строки outbound (только черновик)

**Актор:** администратор фулфилмента.

**Поток**

1. У **черновика** outbound админ удаляет строку.
2. Строка исчезает; если это была последняя строка, привязка продавца к заявке может сброситься; резерв по строке снимается.

**Результат**

- Освобождает остаток для других заявок или перемещений без «фиктивной» отгрузки.

**Ограничения**

- В статусах **submitted** и **posted** удаление строки таким способом недоступно (нужны обычные сценарии отгрузки).

---

## 11. Импорт Wildberries (только чтение)

**Актор:** администратор фулфилмента.

**Предпосылки**

- Есть хотя бы один **продавец** (интеграция на продавца).
- Токены API WB пользователь получает вне приложения (кабинет WB, категории: контент, поставки и т.д.).

**Поток (высокий уровень)**

1. Выбор продавца для интеграции.
2. **Сохранение токенов** (на сервере в зашифрованном виде; в UI только флаги «токен есть», без секретов).
3. **Синхронизация карточек** (первая страница) — фоновая задача; по завершении — краткий итог.
4. **Синхронизация FBW-поставок** (первая страница) — фоновая задача; итог по завершении.
5. Просмотр списков **импортированных карточек** и **импортированных поставок** (снимки в БД).
6. **Привязка внутреннего SKU** к **nm_id** WB (и правила vendor code — как реализовано) для того же продавца, что и у товара.

**Результат**

- Тенант может зеркалить снимки каталога/поставок WB и связать внутренний SKU с идентичностью WB.

**Ограничения (MVP)**

- **Нет** создания карточек или поставок в WB из этого продукта — только импорт/чтение.
- Пагинация, дополнительные поля, печать этикеток — **вне** этого среза.
- В CI HTTP к WB **заглушен**; на боевом сервере нужны валидные токены и сеть наружу к WB (или свой mock).

---

## 12. Учётная запись продавца и «кабинет»

**Актор:** админ создаёт учётку; продавец входит по этому email.

**Поток**

1. Админ создаёт **учётную запись продавца** (email + пароль), привязанную к продавцу.
2. Продавец входит → дашборд в контексте продавца.
3. Продавец видит **отфильтрованные** списки: товары, сводки приёмки/отгрузки, движения, балансы — только данные своего продавца (по правилам бэкенда).
4. Продавец может создавать **черновики** приёмки/отгрузки по своему ассортименту там, где UI разрешает; чувствительные шаги фулфилмента остаются на стороне админа.

**Результат**

- Разделение ролей: самообслуживание продавца в рамках полномочий; фулфилмент сохраняет контроль над исполнением.

---

## 13. Глобальные движения и движения по outbound

**Актор:** любой авторизованный пользователь в рамках своей видимости.

**Поток**

- **Глобальные движения** — обновляемый список последних складских движений (типы: приёмка, перемещение, отгрузка и т.д.).
- **По заявке приёмки / отгрузки** — список движений при выборе заявки.

**Результат**

- Прослеживаемость изменений остатка и времени (в пределах отображаемых лимитов).

---

## 14. Фоновые задачи (общие + WB)

**Актор:** администратор фулфилмента для доступных в UI сценариев.

**Примеры**

- **Сводка по движениям (digest)** — тяжёлая агрегация по журналу; пользователь запускает задачу, опрашивает статус до **done/failed**, читает короткий текст результата.
- **Задачи синхронизации WB** — тот же паттерн: старт, ожидание завершения, чтение итога.

**Результат**

- Долгая работа не выдаётся за синхронный вызов; пользователь видит жизненный цикл.

**Ограничения**

- При настроенном Celery задачи идут на воркерах; без брокера (локально/в dev) может применяться **встроенное/фоновое** выполнение по настройкам деплоя.

---

## 15. Навигация и вёрстка (UX)

**После входа**

- Пользователь видит **навигацию по разделам** (якоря) для перехода к **каталогу** и **операциям** на длинной одностраничной раскладке.
- **Администратор фулфилмента** видит **боковое меню и основную область**: стартовый экран — **дашборд тенанта** с **недельным планирующим видом** (плановые привозы по приёмке и плановые отвозы по отгрузке в смысле «запланировано» для бизнеса) и **краткими списками** последних документов приёмки/отгрузки; в навигации есть пункт **«Поставки и загрузки»** (Supply and Load), открывающий **единый список** (в т.ч. заглушки «выгрузка на МП» / «расхождение» по мере продукта) и те же сценарии карточки документа, что и в разделе **операции** (в рамках объёма продукта).
- Пока грузится профиль (`/auth/me`), показывается **состояние загрузки**, а не наложение форм входа/регистрации друг на друга.
