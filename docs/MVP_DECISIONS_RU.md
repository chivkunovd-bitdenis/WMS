# Решения для старта разработки WMS (зафиксировано)

Документ — **единый источник правды** для автопилота: что строим в MVP, какие интеграции, какие границы. Обновляется только осознанно (через PR).

## Продукт и тенантность

- **Модель**: SaaS — один инстанс → много фулфилментов (тенантов) → у каждого много селлеров.
- **MVP по контуру**: операции склада + **кабинет селлера** + **биллинг v1**.
- **Язык интерфейса**: русский (аккаунт/продукт ориентированы на RU).

## Биллинг v1

- **Хранение**: **литр‑день** — тариф «рубль за литр в сутки» (или копейки; точность в настройках тарифа).
- **Объём**:
  - если у SKU заданы габариты — \(Д \times Ш \times В\) → литры на штуку × количество;
  - иначе — объём **тары/места** (короб/пакет/лот);
  - допускается **ручной override** объёма на позицию/поставку на период онбординга.
- **Граница суток и валюта**: **МСК**, **RUB**, **НДС в MVP не считаем** (отдельный эпик позже).

## Аутентификация

- **Email + пароль** (+ восстановление пароля по плану реализации).
- Роли и изоляция данных обязательны с первых релизов кабинета и мультитенантности.

## Wildberries (первая интеграция)

### Принцип

- **Только импорт (read/sync в нашу систему)**. Создание поставок и карточек **в WB через наш продукт в MVP не делаем** (нет исходящих «create» в сторону WB).

### Методы API (ориентир документации WB)

| Назначение | Метод | Базовый хост (по документации) |
|------------|--------|--------------------------------|
| Список карточек товаров | `POST /content/v2/get/cards/list` | `https://content-api.wildberries.ru` |
| Список поставок FBW | `POST /api/v1/supplies` | `https://supplies-api.wildberries.ru` |
| Детали поставки | `GET /api/v1/supplies/{ID}` | `https://supplies-api.wildberries.ru` |
| Товары поставки | `GET /api/v1/supplies/{ID}/goods` | `https://supplies-api.wildberries.ru` |

### Токены и лимиты

- У WB разные категории доступа (**Контент**, **Поставки** и т.д.) — в продукте хранить секреты **только** в env / secrets, раздельно по тенанту/селлеру.
- Соблюдать лимиты запросов из документации (очередь, backoff, идемпотентность по `requestId` где применимо).

## Терминология (фулфилмент ↔ маркетплейс, RU)

Закреплено для продукта и UI; в коде/API могут оставаться исторические имена (`marketplace_unload`, `outbound` и т.д.) — смысл в интерфейсе и документации **только** по таблице ниже.

| Термин (RU) | Смысл | Кто создаёт / инициатор |
|-------------|--------|-------------------------|
| **Приёмка** (ранее «поставка» в UI) | Товар идёт **от селлера на фулфилмент** (входящий поток на склад ФФ). В портале ФФ — раздел **«Приёмка»**; в коде/API — `inbound` / `inbound_intake`. | **Селлер** создаёт заявку; **ФФ** ведёт приёмку в очереди «Приёмка» → «Сортировка». |
| **Отгрузка** (в смысле «на МП») | Фулфилмент **отгружает на маркетплейс** (склад МП / вывоз на площадку). **Не** называем это «выгрузкой» или *download* в пользовательских текстах. | **ФФ** (операции; документ в системе может называться технически `marketplace_unload` и т.п.). |

Если в сообщении без контекста неясно, о какой цепочке речь — **уточнять у автора**: приёмка (селлер→ФФ) или отгрузка (ФФ→МП). Термин «поставка» в пользовательском UI **не используем** для селлер→ФФ (кроме импорта WB FBW, если явно про склад МП).

## Онбординг селлера (кто и как)

- **Добавляет только админ FF** (раздел «Селлеры» в портале): название бренда + **email**.
- Цепочка API: `POST /sellers` → `POST /auth/seller-accounts` **без пароля** → в БД случайный hash и `must_set_password=true`.
- Админ передаёт селлеру **email** и URL портала селлера; **пароль по API не возвращается**.
- **Первый вход селлера:** email + **пустой пароль** → экран задания пароля → дальше обычный вход.
- UI FF: только MUI, см. `docs/UI_DESIGN_SYSTEM_RU.md`.

## Кабинет селлера (MVP — функции)

Селлер в кабинете должен иметь как минимум:

1. **Создание заявок** (входящие процессы: приёмка / отгрузка — формулировка бизнес‑сущности «заявка» в терминах WMS).
2. **Приём** — сценарий участия/контроля приёмки со стороны селлера (в объёме MVP: статусы, состав, расхождения — по мере готовности бэкенда).
3. **Отгрузка** — создание/отслеживание отгрузки на стороне фулфилмента; синхронизация с импортом WB для FBW — **только отображение импортированных поставок**, без создания в WB из UI в MVP.

## Печать

- **Накладная поставки (приёмка на ФФ):** в документе поставки (приёмка) — кнопка «Печать накладной», A4 в браузере; для сотрудника склада перед приёмкой по составу и коробам.
- **Накладная отгрузки:** отгрузка на МП и operational outbound — отдельные кнопки в соответствующих документах.
- **Этикетки коробов:** термо **58×40**, CODE128 — в блоке коробов приёмки (не путать с накладной).
- Принтер этикеток: **термо 58 мм**, лента **40 мм** (детали ESC/POS/ZPL — в задаче на модуль печати).

## Упаковка и задание на упаковку

Полное описание процесса: **[docs/PACKAGING_RU.md](PACKAGING_RU.md)**.

- **Документ «Задание на упаковку»** — один тип, одна форма; раздел **«Упаковка»** в операциях ФФ. Не этап жизненного цикла приёмки.
- **Остаток в месте** делится на **не упаковано** и **упаковано**; задание переводит количество между ними без смены ячейки.
- **Три входа** (приёмка/сортировка, ручное задание, отгрузка на МП) — один и тот же документ; из сортировки можно упаковать **часть** товара **без** отгрузки.
- **Отгрузка на МП:** в состав добавляют **неупакованный** товар; при статусе **«Начата сборка»** автоматически создаётся **одно** связанное задание (1:1); двусторонние ссылки (как связанные документы в 1С); **провести отгрузку нельзя**, пока задание не выполнено.
- Строка задания: всего / уже упаковано на полке / подтверждено / нужно упаковать / упаковано в работе; на каждой строке — **ТЗ на упаковку** из карточки товара (селлер).
- Подбор по ячейкам **до** заполнения split упаковано/не упаковано; данные split — из остатка в местах подбора.

## Резерв под внутреннюю отгрузку (outbound)

- В `draft` / `submitted`: резерв по строке — **по ячейке**, если ячейка назначена; иначе **по складу ФФ** (суммарно по SKU, без привязки к ячейке).
- **Submit** разрешён без ячейки: предварительный резерв держит доступное на складе.
- **Списание / фактический подбор** — только с назначенными ячейками (`lines_missing_storage` при проведении, если ячейки нет).
- Ячейку можно назначить при добавлении строки, в черновике или при подборе после submit.

## E2E и автопилот

- Каждая фича с UI → минимум один Playwright‑сценарий **пользовательского** пути + стабильные `data-testid`.
- Внешний WB API в CI **мокируется** (фикстуры/Wiremock/записанные ответы); отдельные «живые» смоук‑тесты — вручную или в ночном job с секретами.

## Что сознательно не входит в MVP (по текущим решениям)

- Создание поставок и карточек в WB через API из продукта.
- НДС и бухгалтерия полного цикла.
