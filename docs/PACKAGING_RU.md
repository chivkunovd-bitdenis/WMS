# Задание на упаковку и связь с отгрузкой на МП

Документ фиксирует **согласованное решение** (2026-05-30): один тип документа «Задание на упаковку», раздел «Упаковка» в портале ФФ, двусторонняя связь с отгрузкой на маркетплейс. Источник правды для нарезки issues и реализации.

Связанные документы: [`MVP_DECISIONS_RU.md`](MVP_DECISIONS_RU.md) (краткое резюме), [`USER_STORIES_PRODUCT_RU.md`](USER_STORIES_PRODUCT_RU.md) (US-P, US-G).

---

## 1. Зачем отдельный документ

Упаковка и маркировка — **отдельная работа на складе**, не этап жизненного цикла заявки на приёмку.

Сотрудник должен иметь возможность:

- упаковать товар **сразу после поштучной приёмки**;
- упаковать **часть** товара из зоны «Сортировка» или ячейки **без отгрузки**;
- упаковать товар **в рамках отгрузки на МП**, с явной связью документов (как связанные документы в 1С).

В состав отгрузки добавляют **неупакованный** товар (намерение). Упаковка выполняется в **задании на упаковку**. **Провести отгрузку нельзя**, пока связанное задание не выполнено.

---

## 2. Термины

| Термин | Смысл |
|--------|--------|
| **Задание на упаковку** | Документ работы: что, откуда, сколько упаковать; опциональная связь с отгрузкой на МП. В коде: `packaging_task`. |
| **ТЗ на упаковку** | Инструкция селлера в карточке товара (этикетки, пакет, ЧЗ и т.д.). |
| **Не упаковано / упаковано** | Две части остатка в одном месте (сортировка или ячейка). Сумма = фактический остаток. |
| **Отгрузка на МП** | Документ `marketplace_unload`; пользовательский термин — «отгрузка» (см. `MVP_DECISIONS_RU.md`). |

---

## 3. Остатки

В каждой ячейке и в зоне «Сортировка» для SKU ведётся:

- **quantity_unpacked** — не упаковано;
- **quantity_packed** — упаковано;
- **quantity_on_hand** = unpacked + packed (как сейчас «остаток в месте»).

**Задание на упаковку** при выполнении переводит количество: unpacked ↓, packed ↑ **в том же месте** (адрес не меняется).

Резерв под отгрузку на МП — по **суммарному доступному** (упакованному и неупакованному), пока строка добавлена в состав отгрузки.

---

## 4. Документ «Задание на упаковку»

### 4.1. Один тип, одна форма

Независимо от того, создано задание из приёмки, сортировки или отгрузки — **один тип документа и одна форма**. Различается только заполнение полей и видимость блока «Связанная отгрузка».

### 4.2. Шапка

- номер, статус, склад;
- **связанная отгрузка на МП** (если есть): ссылка «Отгрузка №…» — переход в документ отгрузки;
- опционально: ссылка на приёмку (контекст создания).

### 4.3. Статусы

| Статус | Смысл |
|--------|--------|
| `draft` | Создано, работа не начата |
| `in_progress` | Частично выполнено |
| `done` | Все строки закрыты |
| `cancelled` | Отменено (только без связи с проведённой отгрузкой; правила уточняются в issue) |

### 4.4. Строка задания (единая структура)

| Поле | Описание |
|------|----------|
| Товар | SKU, наименование, фото |
| **Инструкция** | ТЗ на упаковку из карточки товара — **на каждой строке**, всегда видна при работе |
| Место | Ячейка или «Сортировка» |
| **Всего в задании** | Сколько штук в этой строке работы |
| **Уже упаковано на полке** | Система: сколько из подобранного/выбранного остатка относится к `quantity_packed` |
| **Подтверждено с полки** | Оператор подтверждает факт «взял уже упакованные» |
| **Нужно упаковать** | `Всего − Подтверждено с полки` (считает система) |
| **Упаковано в задании** | Факт работы по неупакованной части |
| Готово по строке | `Подтверждено с полки + Упаковано в задании = Всего` |

При создании **из сортировки вручную** поля «уже упаковано / подтверждено» могут быть 0 или отражать смесь в зоне.

## 5. Раздел «Упаковка» (UI ФФ)

Пункт меню **«Упаковка»** в блоке операций:

1. **Очередь** — открытые задания (`draft`, `in_progress`): товар, место, прогресс, пометка «для отгрузки №…» при связи.
2. **Новое задание** — вручную: место → товар → количество.
3. **Карточка задания** — таблица строк, инструкции, ввод прогресса (скан / +N шт.), «Выполнено» по строке и по документу.

Точки входа (все создают или открывают **тот же** документ):

| Откуда | Действие |
|--------|----------|
| Приёмка / сортировка | «Упаковать» → задание с контекстом места и товара → форма упаковки |
| Сортировка / остатки | «Создать задание на упаковку» → частичное количество, **без** отгрузки |
| Отгрузка на МП | «Упаковка» → открыть **связанное** задание (см. §6) |

Форма упаковки — **полноэкранная модалка** или отдельный маршрут с кнопкой «← К отгрузке». Прогресс **сохраняется** при закрытии; повторное открытие — **тот же id**, новое задание не создаётся.

## 6. Отгрузка на МП и связь документов

### 6.1. Состав отгрузки

- В отгрузку добавляют строки (товар, количество) — в том числе **неупакованный** остаток.
- Без заполненного **ТЗ на упаковку** у товара селлер не может отправить заявку (US-G-02).

### 6.2. Подбор по ячейкам

Перед упаковкой по отгрузке должен быть **подбор**: из каких ячеек / «Сортировки» сколько забираем под каждую строку (существующий контур pick allocation для `marketplace_unload`).

### 6.3. Авто-создание задания (решение A)

При переводе отгрузки в статус **«Начата сборка»** система **автоматически** создаёт **одно** задание на упаковку:

- связь 1:1: одна отгрузка — одно активное задание (повторный переход не создаёт второе; открывается существующее);
- строки задания заполняются из **строк отгрузки + подбора по ячейкам**;
- для каждой строки система считает split: сколько из подобранного **уже упаковано** в месте, сколько **не упаковано**.

### 6.4. Подтверждение «уже упаковано с полки»

Оператор видит, например: «30 шт. в задании: 10 уже упакованы в ячейке A, 20 не упакованы». Подтверждает 10. В работе остаётся **упаковать 20**.

Если товар заранее упаковали отдельным заданием из сортировки — при подборе под отгрузку в «уже упаковано» попадёт больше, «нужно упаковать» — меньше или 0.

### 6.5. Навигация (как связанные документы в 1С)

**В отгрузке:**

- блок «Задание на упаковку №…» со статусом и прогрессом (например «15/30»);
- кнопка **«Упаковка»** → форма задания;
- после частичной работы — закрыл форму → в отгрузке снова кнопка/ссылка → **продолжить то же задание**.

**В задании** (если связано):

- ссылка **«Отгрузка №…»** в шапке.

### 6.6. Блокировка проведения

- **Скан в короб отгрузки** и статус **«Отгружено»** — только если связанное задание в статусе **`done`** (по каждой строке: взято с полки + упаковано в задании ≥ количество по отгрузке / подбору).
- Добавление строк в состав отгрузки **не** требует предварительной упаковки.

### 6.7. Изменение подбора

Если подбор по ячейкам **изменили** после начала упаковки:

- при открытии задания **пересчитываются** «уже упаковано / нужно упаковать»;
- уже зафиксированный «упаковано в задании» **не обнуляется** без явного предупреждения оператору.

---

## 7. Что не делаем

- **Не** вводим статус «упакован» в жизненный цикл **заявки на приёмку**.
- **Не** запрещаем добавлять неупакованный товар в **состав** отгрузки.
- **Не** делаем два типа документов или две разные формы для «упаковки из сортировки» и «упаковки из отгрузки».
- **Не** создаём второе задание на упаковку для той же отгрузки, пока первое не отменено (MVP: 1:1).

## 8. Этапы реализации (рекомендация)

| Этап | Содержание |
|------|------------|
| **1** | Остатки `unpacked` / `packed`; миграция; балансы в API и каталоге |
| **2** | Модель `packaging_task` + API; раздел «Упаковка»; ручное задание из сортировки |
| **3** | ТЗ на упаковку в карточке товара; инструкция на строке задания |
| **4** | Связь с отгрузкой на МП: авто-создание при «Начата сборка», навигация, блокировка «Отгружено» |
| **5** | Кнопки входа с приёмки/сортировки; биллинг; ЧЗ (отдельный эпик) |
