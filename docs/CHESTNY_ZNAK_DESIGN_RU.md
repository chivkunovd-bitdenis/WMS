# Честный Знак — ТЗ блока (анализ + архитектура + детальные экраны + фичи)

> Статус: предложение (ТЗ). Цель — превратить текущую «кашу» в самый удобный модуль маркировки:
> минимум кликов, минимум ручного пути, ноль немаркированных отгрузок. Источник правды по блоку.
> Связано с [docs/MVP_DECISIONS_RU.md](MVP_DECISIONS_RU.md), [docs/PACKAGING_RU.md](PACKAGING_RU.md).

## 0. Граница ответственности

**Ядро** — физический склад: принять коды, напечатать, наклеить, проследить правильность по каждой
единице, не отгрузить немаркированное.

**Сверх ядра — value-add автоматизация через API** (фичи-усилители §8.1):
- авто-**ввод в оборот** через API ЧЗ (по факту наклейки/отгрузки);
- авто-**передача КМ в поставку WB/Ozon** (привязка кодов к поставке / УПД через ЭДО);
- опц. **эмиссия** кодов через СУЗ для неотмаркированного товара (маркировка-как-сервис);
- авто-**сверка** статусов кодов с ЧЗ (детектор «мёртвых» кодов).

**Сознательно вне скоупа:** агрегация в короба/паллеты (КиТУ) — для нашего сценария пользы мало,
сложность высокая.

Зависимость для API-части: доступ к ЧЗ/СУЗ/МП от имени селлера (токен/доверенность) — см. §5.9, §10.

## 1. Требования пользователя (как поняты)

1. Коды можно привязать к одному товару **или к группе товаров** (один пул на 20 моделей курток).
2. Либо свой пул на каждый товар (1:1).
3. Загружать коды могут **и селлер, и ФФ**.
4. Учёт **каждого** уникального кода: остаток по пулу/товару/селлеру, карточки и номера; код
   уникален, клеится один раз, списывается.
5. Лента расхода (всё / селлер / товар / документ). Перепечатку испорченного делает только
   **старший смены**, по запросу/подтверждению.
6. Печать при упаковке: гибкий **конструктор ленты** — этикетки и ЧЗ в нужном порядке; по
   умолчанию пара «код на товар + такой же на пакет», дальше следующий код и т.д.
7. Нехватка кодов подсвечивается **сразу** при открытии печати.
8. Уведомления: селлеру — что кодов мало (заранее), ФФ — что созданы отгрузки/приёмки.
9. Списанные коды привязаны к документу упаковки.
10. Человеко-номера документов (`УПАК-26-06-26-1` и т.п.).

Файлы: `backend/app/models/marking_code.py`, `backend/app/services/marking_code_service.py`,

## 3. Ключевые архитектурные решения

### Решение 1. Пул = ЧЗ-карточка; связь «пул ↔ товары WB» задаёт человек вручную (треб. 1, 2)

ЧЗ-код **ничего не знает** про карточки селлера на WB. Селлер в ЧЗ создаёт карточку («Куртка
женская красная» = один GTIN), эмитит под неё 1000 кодов. На WB те же куртки названы как угодно
(SEO-мусор) и это **20 разных карточек** (принт на груди / молния / глянец) — для ЧЗ все они одна
«Куртка женская красная». Сопоставить по названию/коду **нельзя и не нужно** — связь знает только
человек.

- **GTIN автоматически только РАЗБИВАЕТ файл на пулы** (один пул = одна ЧЗ-карточка = один GTIN).
- **Связь «пул ↔ товары WB» — руками** (селлер или ФФ): при загрузке называет пул и отмечает
  галками карточки WMS. Many-to-many. Это момент «вот эти коды = вот эти карточки».
- **Внутри пула коды взаимозаменяемы.** Любой код «красной куртки» подходит на любую из 20
  моделей — упаковщик берёт следующий свободный. Определять «что куда» не нужно.
- **Остаток на пул (общий на группу):** 1000 → упаковали 20 → 980. Показываем «980 на пуле,
  привязан к 20 товарам».

Модель: связь в `marking_pool_products` (M2M), код ссылается на `pool_id`. `product_id` у кода =
факт применения (с какой карточкой упаковали — для прослеживаемости), не ограничение.

### Решение 2. «Код на товаре = код на пакете» — 1 КМ = N этикеток (треб. 4, 6)

Код расходуется **один раз**. «2 экземпляра» = две физические этикетки **одного** кода (на товар +
на пакет, чтобы читался снаружи), а не второй расход. Остаток минус **1 за единицу**, независимо
от копий. `copies` — параметр печати, не инвентаря.

### Решение 3. Журнал событий — единый источник правды (треб. 5, 9)

Таблица `marking_code_events`: каждый переход (загружен, напечатан, перепечатан, наклеен, брак,
заменён, отгружён, аннулирован) с `actor`, `document_number`, `packaging_task_id`, причиной,
временем. Это «лента расхода» (5), привязка к документу (9) и история кода для возвратов.

### Решение 4. Роль «старший смены»

`shift_lead` — флаг-право на `fulfillment_staff`. Подтверждает перепечатку/перемаркировку, видит
очередь запросов, печатает «под собой».

### Решение 5. Жизненный цикл кода (только физический, без ГИС)

```
available → reserved → printed → applied → introduced → shipped → transferred
                │          │         │
                └──────────┴─────────┴──→ defective → replaced (новый код)
                                              └──→ void (аннулирование)
```
- `available` — загружен, свободен (в «Доступно»).
- `reserved` — зарезервирован под строку упаковки (защита от двойной выдачи параллельным упаковщикам).
- `printed` — напечатан, привязан к строке/документу.
- `applied` — наклейка подтверждена сканированием (опц. шаг контроля).
- `introduced` — введён в оборот через API ЧЗ (авто, фича-усилитель §8.1).
- `shipped` — уехал с отгрузкой, выведен из остатка.
- `transferred` — КМ переданы в поставку WB/Ozon (авто, фича-усилитель §8.1).
- `defective` — испорчен (порвали/нечитаемый).
- `replaced` — заменён новым (старый погашен).
- `void` — ручное аннулирование.

«Доступно» = `available`(+`reserved`). «Расход» = `printed/applied/shipped`. «Брак» = `defective`.

## 4. Доменные факты из ресёрча (что учитываем в UX)

- Data Matrix = **GTIN(14) + серийник + криптохвост** (AI `01/21/91/92`, разделитель GS).
- **5–10% брака** на приёмке WB при ручной маркировке без контроля — отсюда фичи 1–3 (скан-печать,
  печать лентой, контроль пары).
- **Несовпадение** кода в документах и на коробе → заворот всей поставки — отсюда фича 3.
- Повреждённая этикетка = «мёртвый груз» → нужна быстрая перемаркировка — фича 6.
- Обычная термопечать **выцветает за 2–3 мес** → рекомендуем термотрансфер; код нельзя
  перекрывать скотчем/мять — учитываем в макете этикетки и профиле принтера.

## 5. Модель данных

### 5.1 `marking_codes` — расширить
`pool_id` (FK→`marking_pools`), `serial` (из `21`), `crypto_tail`, расширенный `status` (§3.5),
`reserved_by_user_id`/`reserved_at`, `applied_at`, `introduced_at`, `transferred_at`, `consumed_at`, `defective_reason`,
`replaced_by_code_id` (self-FK), `packaging_task_id` (денорм.). `product_id` = факт применения.
Индекс пула: `(tenant_id, pool_id, status)`.

### 5.2 `marking_pools` + `marking_pool_products` — НОВЫЕ (треб. 1/2)
`marking_pools`: `id, tenant_id, seller_id, gtin, title, low_stock_threshold, created_at`.
`marking_pool_products`: M2M пул↔товар (ручное сопоставление). Остаток — на пул.

### 5.3 `marking_code_events` — НОВАЯ (журнал/лента, треб. 5/9)
`id, tenant_id, seller_id, code_id, pool_id, event_type, packaging_task_id,
packaging_task_line_id, document_number, actor_user_id, copies, reason, meta_json, created_at`.
`event_type ∈ {imported, printed, reprinted, applied, shipped, defective, replaced, voided}`.

### 5.4 `notifications` — НОВАЯ (треб. 7/8)
`id, tenant_id, recipient_type{user|seller|ff_role}, recipient_id, type,
severity{info|warning|critical}, title, body, link, payload_json, read_at, created_at`.

### 5.5 `marking_reprint_requests` — НОВАЯ (треб. 5)
`id, tenant_id, code_id, packaging_task_line_id, requested_by_user_id, reason,
status{pending|approved|rejected}, resolved_by_user_id, resolved_at, created_at`.

### 5.6 `print_templates` — НОВАЯ (фича 8)
`id, tenant_id, seller_id?, product_id?, name, layout_json, is_default, created_at`.

### 5.7 Нумерация документов (треб. 10)
`document_sequences (tenant_id, doc_type, date, counter)` + `document_number` на упаковке/приёмке/
отгрузке/батчах ЧЗ. Формат `ПРЕФИКС-ГГ-ММ-ДД-N` (`УПАК-26-06-26-1`). Префиксы: `ПРИЕМ`, `ОТГР`,
`УПАК`, `ЗАГРКМ`, `ПЕРЕМАРК`. **Хранить на бэке** (стабильный, ищется, попадает в экспорт селлеру).

### 5.8 Роль
`shift_lead` — boolean-право на `fulfillment_staff`.

### 5.9 `seller_marking_credentials` — НОВАЯ (для усилителей §8.1, подпись §8.2)
`tenant_id, seller_id, cz_token_enc` (ЧЗ/ГИС МТ), `suz_oms_token_enc` (эмиссия), `marketplace`,
`mp_api_key_enc` (WB/Ozon), `mchd_id` + `mchd_valid_until`, `signing_method`
{`ff_kep_mchd` | `seller_cloud` | `manual`}, `edo_route` {`edo_light_roaming_diadoc` |
`diadoc_direct`}, `auto_introduce`, `auto_emit_limit`. Токены/ключи — **шифрованы**.

## 6. ДЕТАЛЬНОЕ ТЗ ПО ЭКРАНАМ

Формат каждого экрана: **навигация · кто видит · что на экране · кнопки и действия (таблица) ·
переходы · состояния/крайние случаи**. UI — единый MUI (эталон `FfProductsCatalogScreen.tsx`),
этикетки 58×40.

### Экран 1. «Честный Знак» — список пулов (заменяет текущую «кашу»)

- **Навигация:** ФФ-портал → меню «Честный Знак»; кабинет селлера → «Честный Знак». Общий компонент.
- **Кто видит:** ФФ-админ/стафф (по всем селлерам, с фильтром), селлер (только свои пулы).
- **Что на экране (сверху вниз):**
  1. Заголовок «Честный Знак» + (для ФФ) выпадающий **выбор селлера**.
  2. Полоса **KPI-плашек:** «Доступно всего», «Расход за 7 дней», «Брак», «Пулы на исходе» (счётчик
     красным, если есть пулы ниже порога).
  3. Кнопка-«primary» **«Загрузить коды»** (открывает Экран 2). Кнопка «Лента расхода» (Экран 4).
  4. Поиск (по названию пула / GTIN / артикулу товара) + фильтр статуса остатка (Все / На исходе / Пустые).
  5. **Таблица пулов**, строка = пул:

  | Колонка | Содержимое |
  |---|---|
  | Пул | Название («Куртка женская красная») + маленький GTIN под ним |
  | Товары | Чипы артикулов (первые 3 + «ещё N»); пусто → жёлтый бейдж «не привязан» |
  | Доступно | число; **красное**, если < порога |
  | Резерв | зарезервировано под активные упаковки |
  | Напечатано | за всё время |
  | Брак | число; клик → отфильтрованная лента |
  | Прогноз | «хватит на ~N дн.» по расходу за 7 дней; «—» если расхода нет |
  | ⋮ | меню строки |

- **Кнопки и действия:**

  | Действие | Что делает | Переход |
  |---|---|---|
  | Клик по строке | Открывает Экран 3 (карточка пула) | → Экран 3 |
  | «Загрузить коды» | Диалог загрузки | → Экран 2 |
  | ⋮ → «Привязать товары» | Диалог мульти-выбора товаров для пула | модалка |
  | ⋮ → «Коды» | Список кодов пула | → Экран 3, вкладка «Коды» |
  | ⋮ → «Лента пула» | Лента, отфильтрованная по пулу | → Экран 4 |
  | ⋮ → «Порог остатка» | Поле «уведомить, когда меньше N» | модалка |
  | Клик по «Пулы на исходе» (KPI) | Фильтрует таблицу по «На исходе» | — |

- **Переходы:** Экран 1 — хаб. Из него во 2/3/4.
- **Состояния:** пустой список → пустое состояние с кнопкой «Загрузить коды»; пул без товаров →
  жёлтый бейдж «не привязан» + быстрый «Привязать»; загрузка → скелетон таблицы.

### Экран 2. Диалог «Загрузить коды» (треб. 3)

- **Кто видит:** селлер (свои), ФФ (с предвыбором селлера).
- **Что на экране:**
  1. (ФФ) селектор селлера, если не выбран на Экране 1.
  2. **Дроп-зона** «Перетащите файлы CSV/PDF/TXT» + кнопка «Выбрать». Мультифайл.
  3. После парсинга — **превью по пулам** (карточка на каждый найденный GTIN):
     «GTIN …123 — распознано 1000 кодов; дубликаты 4; неформат 1». Несколько GTIN в файле →
     несколько карточек.
  4. На каждый пул: поле **«Название пула»** (предзаполняется ранее данным для этого GTIN) +
     **мульти-селект товаров** (поиск по артикулу/названию; ранее привязанные подсвечены).
  5. Итоговая строка: «Будет загружено: 1000 кодов в 1 пул, привязано к 20 товарам».
- **Кнопки:**

  | Кнопка | Что делает | Переход |
  |---|---|---|
  | Загрузить | Создаёт/дополняет пулы, пишет `imported`, документ `ЗАГРКМ-…` | закрывает → toast «Загружено N», обновляет Экран 1 |
  | Отмена | Закрывает без сохранения | — |

- **Состояния:** дубликаты внутри файла и уже существующие — авто-скип, показаны числом; файл без
  валидных кодов → ошибка «не распознано ни одного кода»; пул без товаров — разрешаем загрузить
  (привяжем позже), но бейдж «не привязан».

### Экран 3. Карточка пула (вкладки: Обзор · Товары · Коды · Лента)

- **Навигация:** из Экрана 1 (клик по пулу).
- **Обзор:** название, GTIN, плашки Доступно/Резерв/Напечатано/Брак/Прогноз, история загрузок
  (батчи `ЗАГРКМ-…`: дата, файл, +N кодов, кто).
- **Товары:** список привязанных карточек (артикул, название WB, требует ЧЗ — да/нет). Кнопки
  «Привязать товары», «Отвязать». Подсказка: «Остаток общий на все товары пула».
- **Коды:** таблица уникальных КМ: `маска кода | статус (чип) | дата | кем напечатан | документ |
  «история»`. Фильтр по статусу, поиск по хвосту кода, **«Экспорт CSV»** (в т.ч. «только
  использованные за период» — отдать селлеру для ввода в оборот). Клик «история» → дровер с
  таймлайном кода.
- **Лента:** события только этого пула (= Экран 4 с фиксированным фильтром).

### Экран 4. Лента расхода ЧЗ (журнал, треб. 5/9)

- **Навигация:** Экран 1 → «Лента расхода»; и из карточки пула.
- **Что на экране:** фильтры-чипы сверху (Селлер · Пул/Товар · Документ · Тип события · Период) +
  поиск по маске кода. Таблица:

  | Время | Событие (цветной чип) | Код (маска) | Пул / Товар | Селлер | Документ | Пользователь |
  |---|---|---|---|---|---|---|

- **Действия из строки** (видны по правам): «Перепечатать» (старший смены), «В брак», «Заменить
  новым», «История кода». **«Экспорт»** — выгрузка отфильтрованного (напр. все `shipped` по
  документу `ОТГР-26-06-26-3` → файл селлеру).
- **Состояния:** бесконечный скролл/пагинация; пустой фильтр → подсказка.

### Экран 5. Печать ЧЗ при упаковке — конструктор + скан-печать (треб. 6/7) ★ ядро

- **Навигация:** экран «Упаковка» (`FfPackagingPage.tsx`). Два пути запуска:
  - **A. Скан-печать (фича 1):** в поле фокуса оператор сканирует штрихкод товара → система по
    привязке товар→пул берёт следующий свободный код → сразу печатает по шаблону по умолчанию.
    Ноль кликов. Счётчик «осталось по строке» уменьшается.
  - **B. Кнопка «Печать ЧЗ»** на строке или **«Печать всех ЧЗ»** на всей задаче (фича 2) → диалог-конструктор.
- **Диалог-конструктор, что на экране:**
  1. Шапка: товар/документ, «нужно N», «доступно M в пуле».
  2. **Полоса нехватки (треб. 7):** если M<N — **красный баннер** «Не хватает 18 из 50» + кнопка
     **«Запросить у селлера»** (уведомление, Экран 7) и переключатель «печатать доступные 32».
  3. **Пресеты** (радио-кнопки): «Парами (реком.)» = на единицу `[ЧЗ][ЧЗ]`; «Этикетки + ЧЗ»;
     «Только ЧЗ»; «Только этикетки»; «Свой шаблон».
  4. **Конструктор** (для «Свой шаблон»): горизонтальная лента-дроп — перетаскиваешь блоки
     `Этикетка` и `ЧЗ`, у каждого счётчик копий. Это раскладка **на одну единицу**.
  5. **Предпросмотр сплошной ленты:** показывает первые 2–3 единицы (видно, что внутри единицы код
     одинаковый — на товар и на пакет, а следующая единица — следующий код).
  6. Кол-во единиц (предзаполнено «нужно N»), профиль принтера, «Сохранить как шаблон» (фича 8).
- **Кнопки:**

  | Кнопка | Что делает | Переход |
  |---|---|---|
  | Печать | Резервирует и помечает коды `printed`, шлёт ленту на печать, пишет события + `qty_marking_printed` | toast, закрытие, обновление строки |
  | Печать всех ЧЗ | То же по всем строкам задачи одной лентой в порядке сборки | — |
  | Запросить у селлера | Уведомление о нехватке | — |
  | Сохранить шаблон | Пишет `print_templates` | — |
  | Отмена | — | — |

- **Состояния:** уже печатали → кнопка становится «Перепечатать» (через старшего смены, Экран 6);
  пул пуст → «Печать» заблокирована, виден только «Запросить у селлера»; параллельный упаковщик
  взял те же коды → резерв не даёт пересечься.

### Экран 6. Очередь перепечатки (старший смены, треб. 5)

- **Навигация:** ФФ-портал → «Перепечатки» (бейдж с числом ожидающих). Видит только `shift_lead`.
- **Что на экране:** список запросов: `Время | Кто | Товар | Код (маска) | Причина | Документ`.
- **Кнопки по строке:**

  | Кнопка | Что делает |
  |---|---|
  | Подтвердить и печать | Печатает **тот же** код (если ещё валиден), событие `reprinted` под старшим |
  | Заменить новым | Старый → `defective`+`replaced`, новый из пула → `printed`; событие; остаток корректен |
  | Отклонить | Закрывает запрос с причиной |

- **Как создаётся запрос:** упаковщик на Экране 5/строке жмёт «Брак/перепечать» на конкретной
  единице → `marking_reprint_requests(pending)` → сюда. (Фича 6 — 2 клика, без поиска кода: код
  уже привязан к строке.)

### Экран 7. Колокольчик + центр уведомлений (селлер и ФФ, треб. 8)

- **Навигация:** иконка-колокольчик в шапке (`AuthedAppLayout.tsx` и кабинет селлера), бейдж непрочитанных.
- **Дропдаун:** последние уведомления (иконка severity, текст, время, кнопка-действие). «Показать все» → страница.
- **Типы уведомлений:**
  - Селлеру: «Мало кодов в пуле „Куртка женская красная“: осталось 40 (~1.3 дня)» → кнопка
    **«Догрузить»** (открывает Экран 2 с этим пулом); «ФФ запросил догрузку»; «Коды загружены: N,
    отклонено k».
  - ФФ: «Новая отгрузка на МП от селлера X»; «Новая приёмка»; «Запрос на перепечатку» (старшему).
- **Действие по клику** ведёт прямо в нужный экран (одна кнопка из алерта в дело).

### Экран 8. Дашборд остатков селлера (фича 5)

- **Навигация:** кабинет селлера → «Честный Знак» (верх) или отдельная плашка на главной.
- **Что на экране:** карточки по пулам: `Загружено | Использовано | Доступно | Расход/день |
  Прогноз «хватит до ДД.ММ»`; жёлтый/красный фон при приближении к нулю; кнопка «Догрузить» на карточке.

### Экран 9. Настройки порогов

- На пул (Экран 1 ⋮ или Экран 3): «Уведомлять, когда доступно меньше N» и/или «меньше чем на K
  дней». Кому слать (селлер, и/или ответственный ФФ). Дефолт — глобальный порог тенанта.

## 7. Сквозные сценарии

1. **Загрузка:** кинул файлы → превью по пулам → назвал пул, отметил товары → `imported`,
   `ЗАГРКМ-…`.
2. **Упаковка (быстрый путь):** оператор **сканирует товар** → следующий код печатается парой
   (товар+пакет) сам; счётчик строки −1; событие `printed`, привязка к `УПАК-26-06-26-1`.
3. **Упаковка (весь заказ):** «Печать всех ЧЗ» → одна лента по всем строкам.
4. **Нехватка:** при печати красный баннер «не хватает 18» → «Запросить у селлера» → селлер видит
   в колокольчике, грузит коды.
5. **Брак:** оператор «Брак» на единице → старший смены в очереди «Заменить новым» → события, остаток корректен.
6. **Отгрузка:** проводим отгрузку → коды строк `shipped`, привязаны к `ОТГР-…`; **экспорт
   использованных кодов** селлеру (он вводит в оборот у себя).
7. **Возврат:** скан кода с возврата → история (пул, документ, статус) → при порче «Заменить новым».

## 8. 10 фич, которые реально режут время/клики/ручной путь

| # | Боль (что сейчас долго/много кликов/руками) | Фича | Что экономит |
|---|---|---|---|
| 1 | Оператор ищет товар в списке и жмёт «печать» на каждую единицу | **Скан товара → код печатается сам** (по привязке товар→пул, следующий свободный, пара товар+пакет) | 0 кликов на выбор, нет ошибки выбора, скорость ×2–3 |
| 2 | Печать построчно/поштучно по заказу | **«Печать всех ЧЗ» одной лентой** по всем строкам в порядке сборки | десятки кликов на заказ |
| 3 | Вручную следить, что код на товаре = код на пакете; пересорт → заворот поставки + штраф | **Контроль пары сканером**: сканишь обе наклейки → зелёный/красный | брак на приёмке МП, ручную сверку |
| 4 | О нехватке узнают посреди смены → стоп | **Резерв под заказ + ранняя подсветка нехватки** при создании упаковки + авто-пинг селлеру | простой смены, панику |
| 5 | Селлер не знает, сколько кодов осталось на ФФ, узнаёт когда кончились | **Живой остаток + прогноз «хватит на N дней»** + проактивный алерт + кнопка «Догрузить» | ручные запросы «сколько осталось», аврал |
| 6 | Порвали код: кто разрешит, какой это был код, как зафиксировать | **Перепечатка в 2 клика**: оператор «Брак» → старший подтверждает → тот же/новый код, авто-лог | поиск кода, бумажки, разборки «кто разрешил» |
| 7 | Возврат/спор: где этот код был, на каком заказе | **Скан кода → вся история** (пул, документ, кто/когда, статус) + экспорт по документу | копание в выгрузках |
| 8 | Каждый раз настраивать раскладку печати | **Шаблоны печати по товару/селлеру** с авто-применением | повторную настройку каждый раз |
| 9 | Файлы льют по одному, руками разбираются что к чему | **Мультизагрузка + авто-разбивка по пулам**, маппинг товаров один раз | ручную работу на загрузке |
| 10 | Вручную проверять, всё ли промаркировано перед отгрузкой | **Рабочий список «осталось промаркировать»** — строки без ЧЗ; ничего не уедет немаркированным | ручную сверку, риск отгрузить без кода |

Критерий «крутости» соблюдён: каждая фича убирает конкретный ручной путь / пачку кликов / ожидание.

### 8.1 Фичи-усилители (net-new, сверх требований — «разъебать»)

Опираются на интеграцию с API ЧЗ/СУЗ/МП (включена решением об авто-вводе в оборот).

| # | Фича | Почему разъёб | Нужно |
|---|------|---------------|-------|
| У1 | **Авто-передача КМ в поставку WB/Ozon** (привязка кодов к поставке через API МП / УПД через ЭДО) | Несовпадение/непередача кодов — причина №1 заворота поставок в 2026. Мы единственные отдаём **идеально совпадающий** список, т.к. ведём поштучный учёт «код↔единица↔документ». Селлер не собирает УПД руками. | API WB/Ozon или ЭДО-оператор; маппинг отгрузка→поставка |

### 8.2 Подпись и доверенность (ЭЦП/МЧД) — как делаем легально

Все действия за селлера в ГИС МТ/ЭДО (заказ кодов, ввод в оборот, УПД) **юридически значимы и
требуют УКЭП**. Личный ключ селлера у себя **не держим**. Механика:

- **МЧД (машиночитаемая доверенность)** — основа. Селлер (директор/ИП) один раз подписывает своей
  УКЭП доверенность на ФФ. Дальше ФФ подписывает действия **своей** КЭП + МЧД — ГИС МТ это
  поддерживает. Автоматизация без хранения чужого ключа.
- **Лестница способов подписи** (pluggable signing-service):
  1. `ff_kep_mchd` — ФФ КЭП + МЧД, авто, без участия селлера (У1/У2/У5).
  2. `seller_cloud` — облачная подпись селлера (КриптоПро DSS), подписание по API.
  3. `manual` — ручное «Подписать» в кабинете селлера, MVP-фолбэк, ноль риска хранения ключа.
- МЧД **ограничивает скоуп** действий ФФ → ограничивает ответственность. Каждое подписанное
  действие пишется в журнал (§5.3) со ссылкой на МЧД — полный аудит.

**Нюанс доставки УПД в WB (проверено):** УПД (тип **ДОП**, передача прав на КМ) формируется в
**ЭДО Лайт** (встроен в ЧЗ, бесплатно), но **WB не принимает напрямую из ЭДО Лайт** — нужен
**роуминг ЭДО Лайт → Диадок** (оператор WB). Срок — **3 дня** после приёмки поставки. В конфиге
селлера храним `edo_route` и **трекаем дедлайн** (часть «зелёной цепочки» У2).

> Проверить под селлера: покрытие МЧД для нужных операций ГИС МТ по API, настройку роуминга у его
> оператора, поддержку облачной подписи выбранным УЦ.
