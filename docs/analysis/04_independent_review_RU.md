## Вердикт

- Структура, scope-дисциплина (DEC-001: `outbound_shipment` не трогаем), порядок задач,
  stop-conditions — **хорошо**, можно строить **после** закрытия 🔴 ниже.
- Но `03` пометил остаточными только 3 мелочи (GAP-R01..R03). На деле есть **4 must-fix мины**
  (инвентарь, кросс-модуль с ЧЗ, противоречие правил, неопределённая сущность) и ряд 🟠.
  Это тот же класс, что баги ЧЗ-ревью: «согласовать резерв» без точной механики = builder
  реализует неверно.

## Что хорошо (не переписывать)

- Правильно найден контур `marketplace_unload`, а не `outbound_shipment`.
- CONFLICT'ы названы честно: списание на `ship` vs add-to-box; «Начать подбор» остаётся;
  упаковка не блокирует короба.
- Порядок задач и зависимости логичны; есть stop-conditions и тест-план.

## 🔴 Must-fix ДО сборки (builder реализует неверно без этого)

### R-01. Инвентарная модель «резерв → списание → откат» не специфицирована
**Где:** TASK-004 шаг 2 «согласовать резерв при confirm с частичным списанием» (DEC-006).
**Проблема:** при confirm резервируется весь план. При add-to-box идёт фактическое списание
инкрементами. Нигде не сказано, **как резерв уменьшается** по мере списания. Если резерв остаётся
полным И идёт списание — двойной учёт (резерв + факт). Builder угадает.
**Надо в spec явно (леджер переходов):**
- confirm: `reserved += qty_plan`.
- add-to-box(q): `reserved -= q`, `on_hand -= q` (атомарно, в одной транзакции).
- remove-from-box(q) (TASK-010): `on_hand += q`, `reserved += q` (откат на тот же источник —
  ячейка/зона/агрегат).
- **cancel/abandon отгрузки** (СЕЙЧАС ОТСУТСТВУЕТ ВООБЩЕ): вернуть весь `on_hand` по факту в коробах
  и снять остаток резерва. Без этого — фантомный сток и висящие резервы.
**Severity: HIGH.**

### R-02. Гонки при списании в короб (нет блокировок) — точно повторит баг ЧЗ
**Где:** TASK-004 (`collect_into_box` → `apply_marketplace_unload_pick`), проверка
`available_at_location` затем запись.
**Проблема:** два оператора кладут в короба из одной ячейки/агрегата параллельно: оба читают
`available`, оба пишут → пересписание (oversell) ниже нуля. В ревью ЧЗ это ровно та же находка
(SELECT без `FOR UPDATE`/`SKIP LOCKED`). План про блокировки молчит.
**Надо:** списание под `SELECT ... FOR UPDATE` на строку остатка ячейки/агрегата, перепроверка
доступного под локом; либо условный `UPDATE ... WHERE available >= q` + проверка rowcount.
**Severity: HIGH.**

### R-03. Кросс-модуль с Честным Знаком: «Весь товар упакован» может пропустить немаркированное
**Где:** TASK-007 (галочка «Весь товар упакован» + complete) vs ЧЗ-инвариант «ничего не уедет без
кода» (`assert_packaging_line_marking_done`, уже на main).
**Проблема:** галочка задумана как «пропустить стандартную упаковку, если уже упаковано». Если она
переводит `PackagingTask → done` в обход проверки наклейки ЧЗ — товар, требующий ЧЗ, уедет
немаркированным. Плюс сегодня есть авто-`done` (`_touch_task`/`is_task_complete`): если он остаётся,
явный gate обходится.
**Надо:** при complete (в т.ч. по галочке) всё равно проверять ЧЗ-требование по строкам
(`requires_honest_sign` → коды напечатаны/наклеены); решить судьбу авто-`done` (отключить, чтобы
завершение было только явным). Это пересечение с уже построенным ЧЗ-модулем — builder его не видит.
**Severity: HIGH.**

### R-04. Прямое противоречие: BR-007 (упаковка после complete неизменна) ↔ BR-010 (план всегда
синкается в упаковку)
**Где:** BR-007 + BR-010/DEC-008, TASK-006/TASK-007.
**Проблема:** после «Завершить упаковку» документ упаковки read-only (BR-007). Но если потом
изменить строки плана (добавить товар) — DEC-008 требует «сразу синкнуть в упаковку». В locked
документ синк невозможен. Что главнее — не решено.
**Надо:** правило: либо **запретить** правку плана после завершения упаковки (и в UI, и в API),
либо **переоткрывать** упаковку при изменении плана (сброс `done`). Зафиксировать одно.
**Severity: HIGH (логическая дыра, builder выберет произвольно).**

## 🟠 Should-resolve (риск операционных тупиков / кривого UX)

### R-05. «Зона сортировки» (DEC-005) — это UNKNOWN, а не решение
`03`/GAP-R02 сам пишет «привязать к существующему StorageLocation/zone, не изобретая сущность» — но
КАКОЙ именно сущности нет. Состояние «адресное хранение вкл., но товар ещё не разложен по ячейкам» —
реальное (принят, не разложен), и списывать из него надо из конкретного источника. Builder либо
выдумает сущность (нарушит stop-condition), либо спишет не оттуда. **Надо назвать конкретный
источник в коде до TASK-002.**

### R-06. Нет пути частичной отгрузки (сток < плана) → возможный тупик
DEC-010 убирает `acknowledge_discrepancy`, ship только при `X=Y` (полное распределение). Если
физически товара меньше плана — `X=Y` недостижимо, отгрузку не закрыть никогда. Реальные отгрузки
бывают частичными. **Надо:** явное решение — либо разрешить ship при `X<plan` с подтверждением
(пометить недопоставку), либо разрешить уменьшить план до факта перед ship.

### R-07. Лимит в модалке короба считает только план, а не физический сток
TASK-011/BR-005 ограничивают добавление планом (`plan - в коробах`). Но не показывают и не
ограничивают **физически доступным** (и остатком ячейки при адресном хранении). Оператор упрётся в
provал списания на ровном месте. **Надо:** в модалке «доступно к добавлению» = `min(остаток плана,
физически доступно[/по ячейке])`.

### R-08. Переключение глобального флага при наличии остатков на ячейках
DEC-009 (дефолт вкл.) + toggle. Если у ФФ сток лежит в ячейках и адресное хранение выключают — откуда
списывать (агрегат)? И обратно? Сейчас флаг трактуется как чисто UI-видимость, но он меняет
**источник списания**. **Надо:** правило поведения при toggle с ненулевыми остатками (запретить
переключение при наличии стока в ячейках / миграция остатка в агрегат / и т.п.).

### R-09. Роль селлера в новом флоу размыта
План весь про ФФ (`FfSuppliesShipmentsPage`). У селлера `SellerMarketplaceUnloadDialog`, TASK-012
лишь «согласовать parity». Но завершение упаковки, короба, скан ячеек — это работа на складе ФФ;
селлер физически этого не делает. **Надо:** явно описать, что видит/делает селлер (скорее всего:
только план + статусы, без коробов/ячеек/упаковки-complete).
