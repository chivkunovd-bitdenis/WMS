### S01 — Селлер: черновик и план отгрузки на МП (plan-only)

**Потребность пользователя (зачем пришёл):**
Селлер хочет сообщить фулфилменту, какие товары и в каком количестве нужно отгрузить на маркетплейс (WB). Он не выполняет складские операции — только формирует намерение (план) и отслеживает статус заявки.

**Успех для пользователя (не для системы):**
Селлер создал или открыл черновик отгрузки на МП, добавил товары и количества, отправил заявку (если нужно), видит актуальный статус (черновик / отправлено / подтверждено / отгружено). Не видит коробов, ячеек, упаковки, штрихкодов коробов и кнопок завершения упаковки — это зона ФФ.

**Границы сценария:**
- Входит: создание черновика, выбор товаров из своего каталога, плановые количества, submit заявки, просмотр статуса, RBAC (только свои отгрузки).
- Не входит: короба, scan ячеек, упаковка, печать ШК, финальный ship — всё это только у оператора ФФ (DEC-015).

**Обязательный путь (шаги пользователя):**
1. Войти в кабинет селлера.
2. Открыть создание отгрузки на МП или существующую заявку.
3. Убедиться, что ИП/магазин фиксирован (свой seller_id, без смены).
4. Добавить товары и количества в план.
5. Сохранить/отправить заявку.
6. Просмотреть статус без доступа к складским вкладкам.

**Обязательные данные / действия / статусы на пути:**
- seller_id фиксирован; чужие товары недоступны (BR-001).
- Статусы: draft → submitted (и далее только чтение статуса со стороны селлера).
- Нет UI: «Короба», «Упаковка», «Начать подбор», scan ячеек, «Завершить упаковку», «Печать ШК».

**Типичные продуктовые риски (из 03 + здравый смысл):**
- Селлер видит те же блоки, что ФФ (короба/упаковка) — путаница ролей.
- Селлер может «завершить» складской этап без ФФ.
- Каталог не фильтруется по seller_id.

**Критерий «не готово» (продуктово):**
- Селлер видит или может вызвать операции с коробами, ячейками, упаковкой или ship.
- Не понятно, на каком этапе заявка и что делать дальше (ожидание ФФ).

### S02 — ФФ: создание отгрузки, выбор ИП, план товаров

**Потребность пользователя (зачем пришёл):**
Оператор фулфилмента принимает или инициирует отгрузку на МП от имени конкретного селлера (ИП). Ему нужен документ с планом товаров — без отдельного этапа «подобрать всё в документ».

**Успех для пользователя (не для системы):**
Оператор создал черновик отгрузки, выбрал ИП/магазин, добавил товары и плановые количества. Видит таблицу плана и счётчик «план отгрузки». Связанный документ упаковки уже существует (или видна вкладка упаковки). Нет кнопки «Начать подбор» как отдельного этапа.

**Границы сценария:**
- Входит: создание отгрузки ФФ, фильтр селлера, добавление/редактирование строк плана, синхронизация строк в упаковку (BR-010, DEC-008).
- Не входит: завершение упаковки, короба, ship (другие сценарии).

**Обязательный путь (шаги пользователя):**
1. Открыть «Поставки и отгрузки» → создать отгрузку на МП.
2. Выбрать ИП/селлера (обязательно для ФФ).
3. Добавить товары и количества (picker/scan).
4. Увидеть план и связь с упаковкой (вкладка или индикатор).
5. Убедиться, что нет отдельного «общего подбора».

**Обязательные данные / действия / статусы на пути:**
- `MarketplaceUnloadRequest` status=draft; `seller_id` задан.
- `PackagingTask` создан при create draft (DEC-003), не только при confirm.
- Изменение строк плана → sync в упаковку (DEC-008).
- Счётчик плана без «подобрано в отгрузку».

**Типичные продуктовые риски:**
- Упаковка появляется только после confirm — оператор не видит её на раннем этапе.
- План и упаковка расходятся после edit строк.
- Один dialog без структуры «Товары» — непонятен порядок шагов.

**Критерий «не готово» (продуктово):**
- Нельзя добавить план без выбора ИП (ФФ).

- Остался отдельный этап общего подбора.

### S03 — Настройка «Адресное хранение» в ФФ

**Потребность пользователя (зачем пришёл):**
Администратор фулфилмента решает, работает ли склад с ячейками или только с агрегированными остатками. От этого зависит весь UX приёмки и отгрузки на МП.

**Успех для пользователя (не для системы):**
Админ включил или выключил «Адресное хранение» в настройках ФФ. При выкл. — в отгрузке/приёмке нет полей и scan ячеек; при вкл. — ячейки обязательны (или зона сортировки). Дефолт для нового ФФ — **включено** (DEC-009). При выключении при ненулевых остатках на ячейках — **остатки автоматически переносятся на зону сортировки** (DEC-019, DEC-018); в UI — info «Остатки с ячеек перенесены на зону сортировки».

**Границы сценария:**
- Входит: tenant-флаг, UI checkbox в настройках, влияние на collect API и скрытие UI ячеек.
- Не входит: полная переработка приёмки (только скрытие ячеек при выкл.).

**Обязательный путь (шаги пользователя):**
1. Открыть настройки ФФ (admin).
2. Увидеть переключатель «Адресное хранение» (дефолт вкл.).
3. Сохранить выкл. → в отгрузке нет scan/выбора ячейки.
4. Сохранить вкл. → при добавлении в короб нужна ячейка или зона сортировки.
5. При ненулевых остатках на ячейках — выключение **разрешено**; остатки мигрируют на зону сортировки, админ видит info-сообщение.

**Обязательные данные / действия / статусы на пути:**
- Boolean на Tenant; persists после reload.
- `collect_into_box`: location optional при выкл., обязателен при вкл. (BR-006, DEC-005).
- Зона сортировки = виртуальная StorageLocation приёмки (DEC-018).

**Типичные продуктовые риски:**
- Флаг есть в БД, но UI отгрузки всегда требует ячейку.
- Можно выключить адресное хранение при живых остатках на ячейках — **миграция на зону сортировки** (DEC-019); без миграции — потеря контроля.
- Дефолт выкл. против DEC-009.

**Критерий «не готово» (продуктово):**
- Нет настройки или она не влияет на отгрузку/приёмку.
- Оператор вынужден scan ячейку при выкл. флаге.

**Ссылки на артефакты:**

### S04 — Упаковка: автосоздание, синк, завершение, ЧЗ

**Потребность пользователя (зачем пришёл):**
Оператор ФФ должен упаковать товар (этикетки, Честный знак) **до** раскладки по коробам. Нужен явный контроль: «весь товар упакован» и кнопка «Завершить упаковку».

**Успех для пользователя (не для системы):**
На вкладке «Упаковка» оператор видит задание, синхронное с планом отгрузки. Может напечатать этикетки/ЧЗ, поставить галочку «Весь товар упакован», нажать «Завершить упаковку». После done — упаковка read-only. Галочка **не обходит** проверку ЧЗ для маркированных строк (DEC-014). Правка плана после done → упаковка снова «в работе» (DEC-013).

**Границы сценария:**
- Входит: вкладка упаковки, sync строк, complete endpoint, ЧЗ-gate, reopen on plan edit, сохранение FfPackagingTaskPanel (REQ-013).
- Не входит: добавление в короба (S05), ship (S09).

**Обязательный путь (шаги пользователя):**
1. Открыть документ отгрузки → вкладка «Упаковка».
2. Увидеть строки из плана (после add/edit в «Товары» — sync).
3. Выполнить упаковку (scan/печать ЧЗ при необходимости).
4. Поставить «Весь товар упакован» (опционально) → «Завершить упаковку».
5. Увидеть статус «завершена»; редактирование заблокировано.
6. (Негатив) Попытка завершить без ЧЗ на маркированной строке — блок с сообщением.

**Обязательные данные / действия / статусы на пути:**
- `PackagingTask.status=done` только через явное завершение (не авто `_touch_task`).
- BR-007 / BR-007′: read-only после done; сброс при edit плана.
- BR-016: ЧЗ-gate при complete (DEC-014).

**Типичные продуктовые риски:**
- Упаковка авто-завершается при scan всех строк — нет явного «Завершить».
- Галочка «Весь товар упакован» обходит ЧЗ.
- Упаковка в отдельном dialog, не во вкладке документа.

**Критерий «не готово» (продуктово):**
- Нельзя явно завершить упаковку или после done можно менять состав без переоткрытия.
- Маркированный товар «завершён» без кодов ЧЗ.

### S06 — Добавление в короб и списание остатков

**Потребность пользователя (зачем пришёл):**
Когда товар кладут в короб, склад должен **сразу** отразить факт: остаток уменьшается, резерв корректируется. При ship не должно быть повторного списания.

**Успех для пользователя (не для системы):**
После добавления qty в короб оператор видит состав короба; остаток на ячейке/агрегате уменьшился сразу. При «Отгружено» статус меняется без второго движения inventory. При отмене отгрузки — откат on_hand и резерва (DEC-016, TASK-019). Параллельные операторы не уводят остаток в минус (DEC-017).

**Границы сценария:**
- Входит: collect_into_box inventory, режимы адресного хранения вкл/выкл, remove-from-box откат, cancel откат, ship без double-write.
- Не входит: UI модалки (S05), финальные проверки ship (S09).

**Обязательный путь (шаги пользователя):**
1. Confirm отгрузки (резерв created).
2. Завершить упаковку, создать короба.
3. Добавить товар в короб (с ячейкой или без — по настройке).
4. Проверить: остаток уменьшился; резерв уменьшился на qty.
5. Ship → статус shipped, остаток не списывается повторно.
6. (Альтернатива) Cancel до ship → остаток восстановлен.

**Обязательные данные / действия / статусы на пути:**
- DEC-006: резерв confirm → списание collect → ship только статус.
- DEC-016: `reserved -= q`, `on_hand -= q` в одной транзакции при collect.
- DEC-017: блокировки при списании.
- Выкл. адресное: агрегированный остаток; вкл.: ячейка или зона сортировки (DEC-005, DEC-018).

**Типичные продуктовые риски:**
- Списание только на ship — расхождение коробов и остатков.
- Двойное списание на ship.
- Отмена не возвращает товар из коробов.

**Критерий «не готово» (продуктово):**
- Остаток не меняется при add в короб или меняется дважды при ship.
- Cancel оставляет «висящий» резерв или товар в коробах без отката.

### S07 — Без этапа «общий подбор в документ»

**Потребность пользователя (зачем пришёл):**
Оператор не должен отдельно «подбирать товар в отгрузку» — только план и добавление сразу в короб. Старый UX с «Начать подбор» вводит в заблуждение.

**Успех для пользователя (не для системы):**
В UI документа отгрузки нет кнопки «Начать подбор», модалки ручного подбора и счётчика «подобрано в отгрузку, но не в коробах». Единственный складской путь — «Добавить товары» в конкретный короб.

**Границы сценария:**
- Входит: удаление UI и ручного save pick allocations; allocations только как побочный эффект collect.
- Не входит: детали модалки короба (S05), inventory (S06).

**Обязательный путь (шаги пользователя):**
1. Открыть подтверждённую отгрузку с планом.
2. Убедиться: нет «Начать подбор» / `ff-mp-picking-dialog`.
3. Перейти к коробам → добавить товар только через короб.

**Обязательные данные / действия / статусы на пути:**
- Нет `ff-mp-start-picking`, `ff-mp-picking-dialog` в UI.
- Счётчики: план / распределено по коробам / остаток — без «picked in document».

**Типичные продуктовые риски:**
- Кнопка осталась, но не работает — хуже, чем явное удаление.
- Два параллельных пути: подбор в документ и в короб.

**Критерий «не готово» (продуктово):**
- Оператор может подобрать товар в документ без привязки к коробу как основной путь.

**Ссылки на артефакты:**

### S08 — Счётчики, предупреждения, меню короба

**Потребность пользователя (зачем пришёл):**
Оператор контролирует прогресс: сколько из плана уже в коробах, статус упаковки, предупреждение при недораспределении. Нужны действия с коробом: печать ШК, копия состава, удаление пустого короба.

**Успех для пользователя (не для системы):**
На вкладках видны: план, распределено X из Y (товары, не число коробов), остаток, статус упаковки. При X < Y — предупреждение на «Короба». Меню короба: печать ШК, копировать в новый короб (в пределах плана), удалить только пустой короб (DEC-007). Удаление строки из короба откатывает collect.

**Границы сценария:**
- Входит: счётчики REQ-011, warning DEC-010/012, box menu REQ-010, remove line, delete empty.
- Не входит: финальный ship и acknowledge недопоставки (S09).

**Обязательный путь (шаги пользователя):**
1. Иметь план и частично заполненные короба (X < Y).
2. Увидеть предупреждение на вкладке «Короба».
3. Печать ШК одного короба из меню.
4. Копировать состав в новый короб (не выше плана).
5. Удалить пустой короб; попытка удалить непустой — ошибка.
6. Убрать строку из короба → qty и остаток откатываются.

**Обязательные данные / действия / статусы на пути:**
- Нет счётчика «подобрано в отгрузку».
- Delete box: qty=0 only (BR-011, DEC-007).
- Copy box: новый короб, лимит плана.

**Типичные продуктовые риски:**
- «Распределено» считает короба, а не товары (DEC-002).
- Можно удалить короб с товаром.
- Предупреждение только на ship, не на вкладке.

**Критерий «не готово» (продуктово):**
- Счётчики не соответствуют ТЗ или нет предупреждения при X < Y на «Короба».
- Удаление непустого короба возможно без снятия товара.

**Ссылки на артефакты:**
- 01: Step 7, BR-008, BR-009, BR-011, REQ-010, REQ-011, DEC-002, DEC-007, DEC-010 | 02: TASK-010, TASK-013 | 03: REQ-011 OK | TC: ff-mp-print-waybill e2e

**Ожидаемые точки в коде (из 02, для batch — не проверять на init):**

### S09 — Финальная отгрузка: печать ШК, ship, недопоставка

**Потребность пользователя (зачем пришёл):**
Оператор завершает процесс: проверяет дату отгрузки на МП и склад МП, печатает все штрихкоды коробов, проводит отгрузку. При недопоставке — явное подтверждение, не молчаливый обход.

**Успех для пользователя (не для системы):**
На вкладке «Финальная отгрузка» заданы дата МП и склад МП. Кнопка «Печать всех ШК коробов» печатает все короба. При X < Y — предупреждение и возможность ship с подтверждением недопоставки (DEC-012). Пустые короба удаляются автоматически при ship (DEC-002). Статус → shipped. Нет полей водитель/авто/пропуск (DEC-004).

**Границы сценария:**
- Входит: вкладка финал, print all ШК, ship checks, discrepancy ack, empty box purge, packaging done check.
- Не входит: создание плана, упаковка, наполнение коробов.

**Обязательный путь (шаги пользователя):**
1. Открыть вкладку «Финальная отгрузка».
2. Заполнить/проверить дату отгрузки на МП и склад МП.
3. Нажать «Печать всех ШК коробов».
4. При неполном распределении — увидеть предупреждение → подтвердить недопоставку.
5. Нажать «Отгружено» → статус shipped; пустые короба исчезли.

**Обязательные данные / действия / статусы на пути:**
- `planned_shipment_date`, `wb_mp_warehouse_id` обязательны.
- `assert_unload_packaging_done` перед ship.
- `acknowledge_discrepancy` / аналог для DEC-012.
- Ship не выполняет повторное списание (DEC-006).
- Без полей перевозки (DEC-004).

**Типичные продуктовые риски:**
- Ship заблокирован навсегда при X < Y (против DEC-012).
- Нет «Печать всех ШК».
- Пустые короба остаются после ship.
- Финал в том же dialog без вкладки.

**Критерий «не готово» (продуктово):**
- Нельзя завершить отгрузку при выполненных проверках или ship без подтверждения при недопоставке неочевиден.
- Повторное списание на ship.

**Ссылки на артефакты:**
- 01: Step 8, BR-008′, REQ-012, DEC-002, DEC-004, DEC-011, DEC-012 | 02: TASK-014, TASK-015 | 03: TASK-014 READY | TC: ff-mp-print-waybill e2e, TC-NEW-PKG-03

**Ожидаемые точки в коде (из 02, для batch — не проверять на init):**
- frontend: `FfSuppliesShipmentsPage.tsx` (вкладка «Финал», print all, ship button)
- backend: `marketplace_unload_pick_service.py` (`ship_request`), `marketplace_unload_service.py`
