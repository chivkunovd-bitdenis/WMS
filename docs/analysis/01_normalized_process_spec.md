# Normalized Process Spec

## Goal
- Что меняем: процесс создания **отгрузки на маркетплейс** (WB-подобная модель) — от черновика и выбора ИП до упаковки, коробов, списания с ячеек и финальной отгрузки.
- Зачем: убрать отдельный этап «общего подбора в документ», перенести складское действие в момент добавления товара в конкретный короб; добавить опциональное адресное хранение; связать обязательную упаковку с коробами и финальным шагом.
- Ожидаемый результат: единая цепочка **ИП → план отгрузки → упаковка → короба → добавление товаров (с ячеек при включённом адресном хранении) → печать ШК → завершение отгрузки**.

## Source Requirements
Источник: `process_otgruzka_wb_like_updated.docx` (версия для ТЗ).

Краткий пересказ без нового смысла:

1. **Глобальная настройка ФФ «Адресное хранение вкл/выкл»** — при выкл. скрываются склады/зоны/ячейки и любая логика ячеек в приёмке и отгрузке; при вкл. — полный контур адресного хранения.
2. **Экран 1 — создание отгрузки и план товаров** — выбор ИП/магазина (у селлера фиксирован); добавление товаров и плановых количеств; без отдельного этапа «подобрать весь товар в документ».
3. **Вкладка «Упаковка»** — при создании отгрузки автоматически создаётся связанный документ упаковки с товарами из отгрузки; галочка «Весь товар упакован»; кнопка «Завершить упаковку»; до завершения упаковки нельзя добавлять товары в короба и двигать отгрузку дальше.
4. **Вкладка «Короба»** — создание N коробов сразу; у каждого короба кнопка «Добавить товары»; меню: печать ШК, копирование в новый короб, удаление; товар добавляется сразу в выбранный короб.
5. **Большая модалка добавления в короб** — фото с превью, план/распределено/доступно, скан ячейки→товара (если адресное хранение) или только товар (если выкл.); ручной fallback; лимит по плану отгрузки.
6. **Счётчики** — план отгрузки, распределено по коробам, остаток к распределению, статус упаковки (без счётчика «подобрано в отгрузку»).
7. **Финальный шаг** — дата отгрузки на МП, склад МП, проверки перед завершением, **печать всех ШК коробов** (поля водитель/авто/пропуск **не в scope** — решение владельца 2026-06-27).
8. **Задел под Android/ТСД** — та же механика scan ячейки → scan товара → списание → короб.

## Target Process

### Step 1. Создание черновика отгрузки
- Участник: ФФ или селлер
- Действие пользователя: создаёт новую отгрузку
- Действие системы: создаёт черновик документа отгрузки и связанный документ упаковки (пустой/с товарами по мере добавления — по ТЗ сразу связь)
- Данные: `MarketplaceUnloadRequest` status=draft, `PackagingTask` linked
- Результат: открыт документ отгрузки
- Изменение статуса/состояния: draft

### Step 2. Выбор ИП / магазина
- Участник: ФФ (выбор) / селлер (фиксированный ИП)
- Действие пользователя: выбирает ИП или принимает свой по умолчанию
- Действие системы: фильтрует каталог и остатки по seller_id
- Данные: seller_id на документе
- Результат: видны только товары выбранного ИП
- Изменение статуса/состояния: без смены статуса

### Step 3. Добавление товаров в план отгрузки
- Участник: ФФ или селлер
- Действие пользователя: добавляет товары и плановые количества (список/scan/импорт)
- Действие системы: создаёт строки плана; синхронизирует состав в связанную упаковку
- Данные: `MarketplaceUnloadLine.quantity`
- Результат: таблица плана, счётчик «план отгрузки»
- Изменение статуса/состояния: draft; **без** общего подбора

### Step 4. Упаковка (вкладка)
- Участник: оператор ФФ
- Действие пользователя: упаковывает товар или ставит «Весь товар упакован»; нажимает «Завершить упаковку»
- Действие системы: ведёт задание упаковки (ЧЗ, этикетки); после завершения блокирует редактирование упаковки
- Данные: `PackagingTask.status=done`
- Результат: статус упаковки «завершена»
- Изменение статуса/состояния: packaging done; отгрузка не может перейти к коробам до этого

### Step 5. Создание коробов
- Участник: оператор ФФ
- Действие пользователя: указывает количество коробов, нажимает «Создать короба»
- Действие системы: создаёт N коробов с номерами и ШК
- Данные: N × `MarketplaceUnloadBox`
- Результат: список коробов
- Изменение статуса/состояния: доступно только после Step 4

### Step 6. Добавление товаров в конкретный короб
- Участник: оператор ФФ
- Действие пользователя: «Добавить товары» напротив короба; scan или ручной ввод
- Действие системы:
  - **Адресное хранение вкл.:** scan/выбор ячейки → товар; списание с ячейки (если товар уже распределён по ячейкам) или с **зоны сортировки** (если адресное хранение включили, но товар ещё не разложен по ячейкам).
  - **Адресное хранение выкл.:** ячейка не выбирается и не создаётся; списание с **агрегированного остатка** селлера/склада.
  - Резерв остатка при confirm отгрузки; **фактическое списание — при добавлении в короб** (DEC-006).
  - Обновляет счётчики распределения; не превышает план.
- Данные: `MarketplaceUnloadBoxLine`, движение остатков
- Результат: состав короба, «распределено по коробам X из Y»
- Изменение статуса/состояния: фактическое складское действие в коробе

### Step 7. Проверка коробов и печать ШК (по коробам)
- Участник: оператор ФФ
- Действие пользователя: проверяет состав коробов; может печатать ШК по отдельным коробам
- Действие системы: показывает **предупреждение**, если количество в коробах меньше плана отгрузки (DEC-010); пустые короба допускаются до финала
- Данные: штрихкоды коробов
- Результат: оператор видит нераспределённый остаток и состав коробов
- Изменение статуса/состояния: без смены статуса документа

### Step 8. Финальная отгрузка
- Участник: ФФ
- Действие пользователя: проверяет дату отгрузки на МП и склад МП; нажимает **«Печать всех ШК»**; завершает отгрузку («Отгружено»)
- Действие системы:
  - Повторно предупреждает, если количество в коробах < плана (DEC-010).
  - **Автоматически удаляет пустые короба** (без строк товаров) при проведении отгрузки (DEC-002).
  - Проверяет: ИП, строки плана, упаковка done, распределение по плану, дата и склад МП.
  - Ship **не** выполняет повторное списание остатков (уже списано в короба).
- Данные: `planned_shipment_date`, `wb_mp_warehouse_id`, status=shipped
- Результат: отгрузка завершена
- Изменение статуса/состояния: shipped

## Business Rules

### BR-001. Одна отгрузка — один ИП
- Правило: в документе только товары выбранного seller_id; чужие товары запрещены.
- Уверенность: **HIGH**

### BR-002. План ≠ подбор в документ
- Правило: на первом этапе только плановые количества; отдельного общего подбора в документ нет.
- Уверенность: **HIGH**

### BR-003. Упаковка обязательна перед коробами
- Правило: пока связанная упаковка не завершена — нельзя добавлять товары в коробa; при попытке — «Сначала завершите упаковку товара».
- Уверенность: **HIGH**

### BR-004. Короб — точка складского действия
- Правило: товар добавляется в конкретный короб; **фактическое списание остатка — в момент добавления в короб**; резерв создаётся при confirm отгрузки.
- Уверенность: **HIGH**

### BR-005. Лимит распределения по плану
- Правило: сумма по коробам не превышает плановое количество по каждому товару.
- Уверенность: **HIGH**

### BR-006. Адресное хранение опционально
- Правило:
  - **Выкл.:** ячейки не создаются и не показываются в отгрузке/приёмке; в отгрузке нет выбора/scan ячейки; списание с агрегированного остатка.
  - **Вкл.:** при добавлении в короб указывается ячейка (scan/ручной выбор) **или** зона сортировки, если товар ещё не разложен по ячейкам; после распределения по ячейкам — списание только с ячеек.
  - **Дефолт для новых ФФ:** адресное хранение **включено** (DEC-009).
- Уверенность: **HIGH**

### BR-007. Завершение упаковки необратимо
- Правило: после «Завершить упаковку» документ упаковки нельзя редактировать.
- Уверенность: **HIGH**

### BR-008. Предупреждение о нераспределённом количестве
- Правило: если количество товаров в коробах **меньше** плановой отгрузки — система показывает предупреждение **на вкладке «Короба»** и **повторно при проведении отгрузки**; ship без полного распределения не допускается.
- Уверенность: **HIGH**

### BR-009. Пустые короба при проведении
- Правило: короба без товаров (пустые) **автоматически удаляются** при проведении/завершении отгрузки; не требуют ручного удаления оператором.
- Уверенность: **HIGH**

### BR-010. Синхронизация плана и упаковки
- Правило: при любом add/update/delete строки плана отгрузки состав связанного документа упаковки **сразу синхронизируется** (DEC-008).
- Уверенность: **HIGH**

### BR-011. Удаление короба
- Правило: удалить короб можно только если он **пустой**; чтобы удалить короб с товаром — сначала убрать весь товар из короба (DEC-007).
- Уверенность: **HIGH**

### BR-012. Scope процесса
- Правило: WB-подобный процесс относится **только** к отгрузке на МП (`marketplace_unload`); операционная `outbound_shipment` **не меняется** (DEC-001).
- Уверенность: **HIGH**

- Связанные шаги процесса: Step 6, Step 8
- Связанный код: `NOT_FOUND` (настройка); `storage_location.py`, `FfSuppliesShipmentsPage.tsx`, `FfInboundRequestView.tsx`, `FfSettingsScreen.tsx`
- Статус в коде: **MISSING**
- Как проверить выполнение: при выкл. — нет полей/scan ячейки, collect без `storage_location_id`; при вкл. — текущее поведение с ячейками/зоной сортировки

### REQ-002. Выбор ИП / магазина на первом экране
- Требование: ФФ выбирает ИП; селлер видит только свой ИП без смены.
- Связанные шаги процесса: Step 2, Step 3
- Связанный код: `FfSuppliesShipmentsPage.tsx` (`ff-mp-create-seller-filter`), `SellerMarketplaceUnloadDialog.tsx`

- Как проверить выполнение: создание отгрузки ФФ только после выбора селлера; каталог фильтруется; селлер не может выбрать чужой ИП

### REQ-003. Плановый состав без этапа «общий подбор»
- Требование: убрать кнопку/модалку «Начать подбор» и состояние «подобрано в отгрузку, но не в коробах».
- Связанные шаги процесса: Step 3
- Связанный код: `FfSuppliesShipmentsPage.tsx` (`ff-mp-start-picking`, `ff-mp-picking-dialog`); `pick-options` / `pick-allocations` API

- Как проверить выполнение: в UI нет отдельного подбора в документ; есть только добавление в короб

### REQ-004. Автосоздание и вкладка «Упаковка»
- Требование: **при создании черновика** отгрузки — связанный документ упаковки; вкладка в документе; двусторонняя навигация; товары плана синхронизируются при каждом изменении строк (BR-010).
- Связанные шаги процесса: Step 1, Step 3, Step 4
- Связанный код: `packaging_task_service.ensure_task_for_unload` (сейчас только при confirm); `FfPackagingTaskDialog`
- Статус в коде: **PARTIAL**
- Как проверить выполнение: сразу после create видна упаковка; add line в отгрузке обновляет строки упаковки

### REQ-005. Галочка «Весь товар упакован» и «Завершить упаковку»
- Требование: явная галочка для уже упакованного товара; кнопка завершения; финальный статус без редактирования.
- Связанные шаги процесса: Step 4

### REQ-006. Блокировка коробов до завершения упаковки
- Требование: до завершения упаковки нельзя добавлять в коробa; сообщение «Сначала завершите упаковку товара».
- Связанные шаги процесса: Step 5, Step 6

- Как проверить выполнение: попытка «Добавить товары» в короб при незавершённой упаковке показывает блокирующую модалку

### REQ-007. Создание нескольких коробов сразу

- Связанные шаги процесса: Step 5

### REQ-008. Добавление товаров через модалку напротив короба
- Требование: у каждого короба кнопка «Добавить товары»; большая модалка (~2×) с фото, планом, распределено, scan/ручной ввод; лимиты по плану.
- Связанные шаги процесса: Step 6
- Связанный код: inline scan в `FfSuppliesShipmentsPage.tsx`; `ProductPhotoThumb.tsx` не используется в блоке коробов

- Как проверить выполнение: модалка открывается из строки короба; фото с hover-zoom; нельзя превысить план

### REQ-009. Списание при добавлении в короб
- Требование: резерв при confirm; **фактическое списание при добавлении в короб** (не на ship). При адресном хранении — с ячейки или зоны сортировки; при выкл. — с агрегированного остатка.

### REQ-010. Действия с коробом (ШК, копия, удаление)

### REQ-011. Счётчики плана и распределения

## Gaps / Contradictions

### GAP-001. Нет переключателя адресного хранения
- Тип: **SYSTEM**
- Проблема: ТЗ требует глобальный флаг ФФ; в коде ячейки всегда в модели и обязательны при сборке.
- Почему важно: без флага нельзя упростить процесс для ФФ без адресного склада.
- Блокирует следующий шаг: **YES**

### GAP-002. Списание остатков отложено до «Отгружено»
- Тип: **PROCESS**
- Проблема: ТЗ — списание при добавлении в короб; код — при `ship_request`.
- Почему важно: расхождение фактического остатка и состава коробов; риск двойного резервирования.
- Блокирует следующий шаг: **YES**

### GAP-003. Сохранён этап «Начать подбор»
- Тип: **PROCESS**
- Проблема: отдельная модалка подбора и блок `pick_allocations` противоречат новой модели.
- Почему важно: ключевое изменение ТЗ (§17) — замена общего подбора на добавление сразу в короб.
- Блокирует следующий шаг: **YES**

### GAP-004. Упаковка не блокирует сборку в коробa
- Тип: **PROCESS**
- Проблема: `packaging_not_done` только на ship; коробa доступны до завершения упаковки.
- Почему важно: нарушает BR-003 и §4.6 ТЗ.
- Блокирует следующий шаг: **YES**

### GAP-005. UI не соответствует вкладочной модели WB
- Тип: **SYSTEM**
- Проблема: один dialog вместо вкладок Товары / Упаковка / Коробa / Финал; упаковка — отдельный dialog.
- Почему важно: UX-контракт ТЗ и порядок экранов (§16).
- Блокирует следующий шаг: **NO**

### GAP-006. Модель «один открытый короб»
- Тип: **PROCESS**
- Проблема: текущий API не поддерживает N коробов с независимым «Добавить товары».
- Почему в important: оператор WB работает с несколькими коробами параллельно.
- Блокирует следующий шаг: **YES**

## Decisions (закрыто 2026-06-27)

| ID | Решение |
|----|---------|
| **DEC-001** (Q-001) | Scope: **только отгрузка на МП** (`marketplace_unload`). `outbound_shipment` не трогаем. |
| **DEC-002** (Q-002) | **Пустые короба** автоматически **удаляются при проведении отгрузки**. «Распределено по коробам» = сколько **товаров** из плана уже в коробах (X из Y), не число пустых коробов. |

| **DEC-005** (Q-005) | **Выкл. адресное хранение:** нет ячеек, нет выбора ячейки, списание с **агрегированного остатка**. **Вкл.:** до разложения по ячейкам — **зона сортировки**; после — списание с ячеек. |
| **DEC-006** (Q-006) | Резерв при **confirm** → **списание при добавлении в короб** → ship только статус и проверки. |

| **DEC-008** (Q-008) | Изменение строк плана → **сразу sync** в документ упаковки. |
| **DEC-009** (Q-009) | Дефолт «Адресное хранение» для ФФ: **включено**. |
| **DEC-010** (Q-010) | Если в коробах меньше товара, чем в плане — предупреждение **на вкладке «Короба»** и **при ship**; ship без полного распределения **нельзя**. |
| **DEC-011** (Q-011) | На финале — кнопка **«Печать всех ШК коробов»**. |

## Decisions v2 — правки независимого ревью (04, 2026-06-27)

Перекрывают конфликтующие пункты выше. Источник: `04_independent_review_RU.md` + ответы владельца.

| ID | Решение |
|----|---------|
| DEC-012 | **Недопоставка разрешена.** Ship при `распределено X < план Y` допускается с явным подтверждением и пометкой недопоставки. План = «заказано», факт = что в коробах. Перекрывает DEC-010 в части «только X=Y»: предупреждение остаётся, ship не блокируется намертво. |
| DEC-013 | **Правка плана после завершения упаковки → авто-переоткрытие упаковки.** Изменение строк плана сбрасывает `PackagingTask` из `done` обратно в работу и пере-синкает состав. Т.к. ship требует завершённой упаковки, оператор обязан завершить упаковку заново. Снимает противоречие BR-007 ↔ BR-010. |
| DEC-014 | **«Весь товар упакован» НЕ обходит ЧЗ.** Галочка завершает упаковку, но для строк `requires_honest_sign` сохраняется проверка: коды ЧЗ должны быть в системе (наклеены/напечатаны прошлым заданием упаковки — валидный сценарий пред-упакованного товара). Если по такой строке кодов нет — завершение блокируется с понятным сообщением. |
| DEC-015 | **Селлер — только план + статусы.** Селлер выбирает товары/количества и видит статусы. Короба, ячейки, завершение упаковки, ШК — только ФФ. `SellerMarketplaceUnloadDialog` без вкладок коробов/упаковки. |
| DEC-016 | **Инвентарный леджер (точные переходы).** confirm: `reserved += qty_plan`. add-to-box(q): `reserved -= q`, `on_hand -= q` (одна транзакция). remove-from-box(q): обратный откат на тот же источник. **cancel/abandon отгрузки:** вернуть весь `on_hand` из коробов + снять остаток резерва. |
| DEC-017 | **Анти-гонка при списании.** Списание из ячейки/агрегата — под `SELECT … FOR UPDATE` либо условным `UPDATE … WHERE available >= q` с проверкой rowcount. Без двойного списания/ухода в минус при параллельных операторах. |
| DEC-018 | **«Зона сортировки» = виртуальная `StorageLocation` приёмки** на складе (источник списания при адресном хранении вкл., пока товар не разложен по ячейкам). Новую сущность не создавать. *(Подтвердить под модель склада.)* |
| DEC-019 | **Toggle адресного хранения блокируется** при ненулевом остатке на ячейках. Сначала обнулить/мигрировать остаток. |
| DEC-020 | **Лимит добавления в короб = `min(остаток плана, физически доступно[/по ячейке])`.** Модалка показывает оба ограничения. |
