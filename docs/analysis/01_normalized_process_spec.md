# Normalized Process Spec

## Goal
- Что меняем: процесс **отгрузки на маркетплейс** (документ `MarketplaceUnloadRequest`) — статусы, UI документа, параллельные потоки **сборки в короба** и **упаковки**, scan готовых коробов с приёмки, единый конструктор печати на вкладке упаковки.
- Зачем: убрать лишние вкладки и блокировки, разделить «сборку с ячеек» и «упаковку по плану», опереться на цепочку **ячейка → короб → товар** из приёмки, дать оператору гибкую печать этикеток/ЧЗ одной лентой.
- Ожидаемый результат: после **«Утверждено»** сотрудник параллельно собирает короба (в т.ч. scan готовых коробов) и ведёт упаковку по всему плану; незавершённая упаковка **не мешает** коробам; финальная отгрузка возможна только при выполненных проверках (упаковка, короба, списание).

## Source Requirements
Источник: уточнения владельца продукта, итерация **2026-06-27** (включая последние пункты по приёмке, переполнению короба и конструктору печати).

Краткий пересказ без нового смысла:

### Документ и статусы
1. Селлер создал план → статус **«На утверждении»** (`submitted`). ФФ нажимает **«Утвердить»** → **«Утверждено»** (`confirmed`). Если отгрузку создаёт ФФ — после утверждения сразу **«Утверждено»**, без промежуточного «на утверждении».
2. Отдельного статуса «упаковка» на документе отгрузки **нет**.
3. **«На сборке»** (`collecting`) — при **первом созданном коробе отгрузки** или **первом добавленном в отгрузку товаре** (что наступит раньше).
4. **«Отгружено»** (`shipped`) — после финального шага, когда остаток списан.
5. Склад FF и дата отгрузки — **в шапке** документа с момента создания; кнопка **«Утвердить»** — **внизу** экрана, как у других документов.
6. Вкладки **«Финальная отгрузка»** и **«Короба»** — **убрать**. Остаются **«Товары»** и **«Упаковка»**. Короба и финальные действия — на основной вкладке / в footer.
7. Плашка прогресса упаковки — **не показывать на черновике** (до сохранения/утверждения). Кнопка **«Продолжить упаковку»** — **не в общем баннере**, только контекст вкладки упаковки.

### Упаковка
8. Задание на упаковку создаётся при **«Утверждено»** на **весь пул товаров** строк плана отгрузки.
9. Упаковку можно выполнять **в любой момент** — до, во время или после сборки коробов; порядок **не регламентируется**.
10. **Незавершённая упаковка не блокирует** создание и наполнение коробов отгрузки.
11. Складское «упаковано на полке» **не используется**. Колонку **«На полке упак.»** в UI упаковки — **убрать**.
12. Упаковано / неупаковано считается **только внутри задания упаковки**: сколько оператор нажал «Упаковать» по строкам. Пример: 100 брюк + 100 юбок; все брюки упакованы → в задании осталось **100 неупакованных** (юбки).
13. Действие «Упаковать» — **прогресс задания**, **без** проверки и движения остатков по ячейкам склада.
14. Финальная отгрузка (**«Отгружено»**) — только при **завершённом** задании упаковки и выполненных требованиях по ЧЗ (где применимо).

### Сборка в короба
15. **Ячейки** участвуют **только** в сборке (наполнении коробов отгрузки), **не** в упаковке.
16. На вкладке **«Товары»**: создание коробов, списки, печать ШК; **строка scan только для короба** (WHB) — сосканировать готовый короб **в отгрузку**; **ячейки и товары эта строка не читает**.
17. **Большая модалка** наполнения короба: ручной выбор «с какой ячейки какой товар»; **строка scan** понимает **ячейку, товар и короб**:
    - scan **ячейки** → последующие товары списываются **с этой ячейки** в открытый короб отгрузки;
    - scan **товара** → добавление в короб с учётом активной ячейки;
    - scan **готового короба** → весь привязанный состав в отгрузку (см. п. 18–19).
18. **Приёмка по коробам**: хранить связь **ячейка → короб → товар** (товар принят/разложен в короб на ячейке). Scan **ячейки + короба** или **только короба** → весь привязанный к коробу товар добавляется в отгрузку.
19. При добавлении **целого короба**: модалка **«Весь короб будет добавлен в отгрузку»** (OK / Отмена). Если в коробе **больше**, чем нужно по плану — дополнительное предупреждение; при **OK** весь товар из короба **идёт в отгрузку**, **включая количество больше плана**.
20. Короб, добавленный scan-ом готового короба, в отгрузке **неотличим** от короба, наполненного вручную по строкам.

### Печать (вкладка «Упаковка»)
21. Иконка печати **только на вкладке упаковки** → **всегда умный конструктор** (не простой диалог 58×40).
22. Блок **ЧЗ** в конструкторе — **только** если на товаре стоит признак ЧЗ; иначе не показывается.
23. Поле **«сколько этикеток на каждый товар»** (число); без расшифровки «короб/пакет/бирка».
24. Конструктор задаёт **порядок и состав ленты** (например: 3 блока ЧЗ подряд, затем 2 этикетки WB — или иначе). После **OK** формируется **вся лента** строго по правилам, заданным в конструкторе, с учётом количества к упаковке и **этикеток на товар**.

## Target Process

### Step 1. Создание и план отгрузки
- Участник: Селлер или ФФ
- Действие пользователя: склад FF, дата, товары, количества
- Действие системы: документ `draft`; у селлера после отправки — `submitted`
- Результат: черновик или «На утверждении»
- Изменение статуса/состояния: `draft` / `submitted`

### Step 2. Утверждение
- Участник: ФФ
- Действие пользователя: «Утвердить» внизу экрана
- Действие системы: резерв; `PackagingTask` на весь план
- Результат: «Утверждено»
- Изменение статуса/состояния: `confirmed`

### Step 3. Сборка (параллельно с упаковкой)
- Участник: Кладовщик
- Действие пользователя: короба; большая модалка (pick/scan cell/product/box); scan WHB на «Товарах»
- Действие системы: pick allocations; attach готового короба с confirm; → `collecting` при первом коробе или товаре
- Результат: товар в коробах отгрузки
- Изменение статуса/состояния: `confirmed` → `collecting`

### Step 4. Упаковка (параллельно со сборкой)
- Участник: Упаковщик
- Действие пользователя: конструктор печати; «Упаковать»
- Действие системы: прогресс в task; **не блокирует** короба
- Результат: упаковка по плану
- Изменение статуса/состояния: task `in_progress` / `done`

### Step 5. Финальная отгрузка
- Участник: ФФ
- Действие пользователя: «Отгружено»
- Действие системы: проверка упаковки и ЧЗ; списание
- Результат: «Отгружено»
- Изменение статуса/состояния: `shipped`

## Business Rules

### BR-001. Упаковка не блокирует короба
- Правило: незавершённое задание упаковки **не запрещает** create/collect коробов.
- Уверенность: **HIGH**

### BR-002. Упаковка без ячеек и «полки»
- Правило: прогресс только в задании упаковки; колонку «На полке упак.» не показывать.
- Уверенность: **HIGH**

### BR-003. Ячейки только при сборке
- Правило: scan ячейки и списание с ячейки — только при наполнении коробов отгрузки.
- Уверенность: **HIGH**

### BR-004. Attach короба — весь состав
- Правило: scan короба с привязанным товаром (цепочка приёмки) добавляет весь состав после confirm.
- Уверенность: **HIGH**

### BR-005. OK при переполнении — весь короб
- Правило: при OK весь товар из короба, **включая сверх плана**.
- Уверенность: **HIGH**

### BR-006. Scan на «Товарах» — только короб
- Правило: главная scan-строка распознаёт только WHB.
- Уверенность: **HIGH**

### BR-007. Ship требует упаковку
- Правило: ship только при завершённом task и ЧЗ где требуется.
- Уверенность: **HIGH**

### BR-008. Task при confirm
- Правило: `PackagingTask` только при `confirmed`, на полный план.
- Уверенность: **HIGH**

### BR-009. Лента по конструктору
- Правило: после OK полная лента по layout конструктора, qty к упаковке и этикеток на товар.
- Уверенность: **HIGH**

## Gaps / Contradictions

### GAP-001. Упаковка блокирует короба
- Тип: **PROCESS**
- Проблема: `packaging_not_done` на create/collect vs BR-001.
- Почему важно: блокирует параллельный процесс «короб + упаковка».
- Блокирует следующий шаг: **YES**

### GAP-002. Упаковка и ячейки
- Тип: **DATA**
- Проблема: `record_pack_progress` + «На полке» завязаны на `InventoryBalance` по ячейке.
