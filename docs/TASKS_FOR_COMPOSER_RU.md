# Задача 1 — Импорт коробов из файла «Штрих-код комбайн» (приёмка + отгрузка на МП)

## Зачем

Клиент собирает короба физически на складе (далеко от компьютера): сканирует товары
телефоном в программу «Штрих-код комбайн», раскладывая по коробам 1, 2, 3… Программа
выгружает xlsx. Сейчас содержимое переносят в WMS руками. Нужно загружать этот файл
напрямую: система по ШК находит товары, подтягивает количества и **сразу раскладывает по
коробам**. Кнопка «Загрузить по накладной» — в приёмке и в отгрузке на маркетплейс.

## Формат файла (строгий)

Эталон: `Список_штрих_кодов_ИП_Фадин_Тула_30_06_2026_11111.xlsx`.

- Лист один (в примере «Штрих-коды», но **на имя листа не завязываться** — берём первый).
- Строка 1 — заголовки, ровно 3 колонки (регистр/пробелы игнорировать, сравнивать по
  нормализованному значению):
  - `Штрих-код`
  - `Кол-во`
  - `Адрес`
- Строки 2..N — данные:
  - **Штрих-код** — EAN-13 WB (напр. `2047892455639`), строка. Может встречаться в
    нескольких строках (в разных коробах).
  - **Кол-во** — целое ≥ 1.
  - **Адрес** — идентификатор короба (в примере `1`, `2`, `3`). Одинаковый «Адрес» = один
    короб. Одна строка = «в короб `Адрес` положить `Кол-во` штук товара с этим ШК».

Пример:
```
Штрих-код       Кол-во  Адрес
2047892455639   2       1
2047892573616   1       1
...
2047892455622   2       2
...
2044125222556   1       3
```
→ короб 1: 8 позиций, короб 2: 15 позиций, короб 3: 16 позиций.

## Разбор и валидация (бэкенд)

Парсинг делаем на бэкенде (надёжнее и единообразно с существующим импортом маркировки).

- Добавить зависимость `openpyxl` в `backend/pyproject.toml` — сейчас бэкенд парсит только
  CSV/TXT/PDF (`parse_import_file`, `backend/app/services/marking_code_service.py:548`),
  xlsx не поддерживается. Читать `read_only=True, data_only=True`.
- Принимать `.xlsx`. Другой тип → ошибка `unsupported_file_type`.
- Валидация структуры → при провале вернуть понятную ошибку и НЕ применять ничего:
  - нет требуемого столбца → `missing_column: <имя>`;
  - лист пустой / нет строк данных → `empty_file`;
  - `Кол-во` не целое/≤0 → ошибка по строке (номер строки);
  - пустой `Штрих-код` или пустой `Адрес` → ошибка по строке.
- Резолв товара по ШК: тот же принцип, что `resolveProductIdByBarcode`
  (`frontend/src/utils/resolveProductByBarcode.ts`) — точное совпадение по SKU, primary
  WB-штрихкоду или любому ШК карточки; по каталогу товаров селлера этой накладной. ШК не
  найден → строка в список ошибок (`barcode_not_found`), остальное грузится.

### Контракт (двухшаговый поток, по образцу импорта маркировки: `/import/preview` + `/import`)

1. `POST .../import-boxes/preview` — принимает файл, возвращает:
   - `boxes`: `[{ address, lines: [{ barcode, product_id, sku_code, product_name, quantity }], total_qty }]`;
   - `errors`: `[{ row, barcode, code, message }]`;
   - `summary`: `{ boxes_count, positions, total_units, error_count }`.
   Ничего не создаёт. Фронт показывает предпросмотр + ошибки.
2. `POST .../import-boxes/apply` — создаёт короба и раскладывает товары. Параметр
   `ignore_errors: bool` (по умолчанию false): при ошибках уровня строк и false → 422 с
   `errors`; при true → грузим распознанное, ошибки возвращаем в ответе для отчёта.

## Раскладка по коробам

**Уже созданные в накладной короба не трогаем — импорт ДОБАВЛЯЕТ новые короба. Никаких
«закрытий» короба: в системе такого понятия нет, ничего закрывать/финализировать не надо.**

### Приёмка (`inbound_intake`)

Модель есть: `InboundIntakeBox` + `InboundIntakeBoxLine`. Короб создаётся `create_open_box`;
количество товара в коробе ставится `set_product_quantity_in_open_box`
(`PUT /operations/inbound-intake-requests/{request_id}/boxes/{box_id}/lines/{product_id}`,
`backend/app/api/inbound_intake.py:767`).

Алгоритм apply:
- для каждого уникального `Адрес` → создать короб (`create_open_box`), запомнить маппинг
  `Адрес → box_id`; если у `InboundIntakeBox` есть поле метки/имени — записать туда `Адрес`;
- для каждой строки → `set_product_quantity_in_open_box(box_id, product_id, quantity)`;
- дубликат (ШК, Адрес) в файле → суммировать количества до записи (учесть, что set
  «устанавливает», а не «добавляет»).

Кнопка «Загрузить по накладной» — в `frontend/src/screens/ff/FfInboundRequestView.tsx`.

### Отгрузка на маркетплейс (`marketplace_unload_requests`)

Модель: короб — `create_marketplace_unload_box`
(`.../boxes`, `backend/app/api/marketplace_unload_requests.py:1198`); товар в короб —
`manual_marketplace_unload_box_line`
(`.../boxes/{box_id}/manual-line`, `:1327`). Проверить, нужно ли параллельно наполнять
план отгрузки (`/lines`, `add_marketplace_unload_line`) или короба сами агрегируются —
посмотреть, как это делает ручная сборка короба, и повторить ту же связку.

Алгоритм apply — тот же: `Адрес → box`, строки → `manual-line` (product + qty).

Кнопка — в `frontend/src/screens/ff/FfSuppliesShipmentsPage.tsx`.

## Крайние случаи

- Дубликат (ШК, Адрес) → суммировать.
- Один ШК в разных коробах → раскладывается по каждому отдельно (норма).
- Товар другого селлера / не из каталога накладной → в ошибки, весь импорт не роняем.
- Пустые строки в конце листа → игнорировать.
- Числа как float (`2.0`) → к int; дробные → ошибка строки.
- Большой файл (сотни строк) → один транзакционный apply; резолв ШК батчем по каталогу,
  не по запросу на строку.

## Тесты

Backend:
- парсер: корректный файл → структура boxes/lines; нет столбца → `missing_column`;
  не-xlsx → `unsupported_file_type`; кривое `Кол-во` → ошибка строки; неизвестный ШК → в
  `errors`, остальное распознано.
- apply (приёмка): создаёт N коробов, раскладывает количества, суммирует дубликаты;
  существующие короба не изменены; `ignore_errors=false` при ошибках → 422; `true` → грузит
  распознанное.
- apply (отгрузка МП): аналогично на модели marketplace unload.
Frontend e2e:
- загрузка примера-файла в приёмку → короба и позиции появились; файл без столбца → ошибка
  формата, ничего не создано.

## Критерии приёмки

1. На страницах приёмки и отгрузки на МП есть кнопка «Загрузить по накладной», принимающая
   строго xlsx указанного формата.
2. По файлу создаются короба (1 короб = 1 `Адрес`), товары находятся по ШК и
   раскладываются с количествами.
3. Ошибки (нет столбца, не тот формат, ШК не найден, кривое количество) показываются;
   при ошибках формата уровня файла импорт не применяется.
4. Существующие короба накладной остаются без изменений — импорт только добавляет новые.

# Задача 2 — ЧЗ: печать из общих корзин + гарантированное списание КМ

Один связный блок: сделать каталожную печать ЧЗ списывающей (это же чинит печать из общих
корзин) и убедиться, что КМ списывается ВЕЗДЕ.

## 2A. Проблема «не хватает N из N» при печати из общих корзин

При печати ЧЗ из вкладки «Честный знак» кнопка блокируется «не хватает 1 из 1», хотя в пул
загружены сотни КМ. Воспроизводится, когда один GTIN привязан к нескольким товарам
(типовой кейс WB: один штрихкод на несколько размеров/цветов) — КМ группируются по GTIN в
пул, а пул с ≥2 товарами считается «общей корзиной».

**Корень:** гейт печати считает `personal_available` — только пулы, привязанные ровно к
ОДНОМУ товару; общие корзины исключены
(`backend/app/services/marking_code_service.py:1186-1215`;
`backend/app/api/marking_codes.py:515-548`). А сам эндпоинт печати `list_product_codes` →
`_code_filter_for_product` (`marking_code_service.py:959-973`) берёт КМ из **всех** пулов
товара, включая общие. То есть коды печать видит, а кнопку до них не пускают.

Точки передачи гейта в модалку:
- `frontend/src/screens/shared/HonestSignScreen.tsx:683` — `markingAvailable: row.personal_available`;
- `frontend/src/utils/useFfProductMarkingPrint.tsx:63-72` (`sumPersonalAvailable`).

Согласованное решение — **разрешить печать из общих корзин**.

## 2B. Критично: каталожная печать сейчас НЕ списывает КМ

Печать из каталога (`source === 'catalog'`, без `lineId`) в
`frontend/src/components/MarkingPrintDialog.tsx` (`printCatalogTape`) делает
`GET /operations/marking-codes/products/{id}/codes`, фильтрует `available`, берёт первые N
и печатает на клиенте. На бэкенд ничего не уходит — КМ остаются `available`. Поэтому:
- повторная печать выдаёт те же КМ;
- для общей корзины товар A и товар B напечатают **одни и те же** коды → один КМ ЧЗ на
  двух физических единицах. Для ЧЗ недопустимо (код уникален, наносится один раз).

**Эталон списания** — печать из строки упаковки `print_codes_for_packaging_line`
(`marking_code_service.py:1345`): выбор `STATUS_AVAILABLE` по `_code_filter_for_product`
(включая общие корзины) с `.order_by(created_at).limit(target).with_for_update(skip_locked=True)`,
перевод в `STATUS_PRINTED`, `record_event(EVENT_PRINTED)`. **Путь упаковки не трогаем — он
корректен, берём за образец.**

## Реализация

### Часть A — бэкенд: новый списывающий эндпоинт печати по товару

`POST /operations/marking-codes/products/{product_id}/print` с телом
`{ quantity: int, layout_json?: PrintLayout, allow_partial?: bool }`. Сервисная функция —
по образцу `print_codes_for_packaging_line`, но без `PackagingTaskLine`:
- проверки: продукт существует, `requires_honest_sign`, есть `seller_id`;
- выбор: `_code_filter_for_product(product_id)` (включает общие корзины) +
  `status == STATUS_AVAILABLE` + `seller_id == product.seller_id`,
  `.order_by(created_at).limit(quantity).with_for_update(skip_locked=True)`;
- `shortage = max(0, quantity - len(codes))`; при `shortage > 0 and not allow_partial` →
  откат и `quantity: 0, shortage` (тот же контракт, что у упаковки);
- перевод КМ в `STATUS_PRINTED`, `record_event(EVENT_PRINTED)` (без `packaging_task`,
  `document_number=None`);
- ответ той же формы, что `_print_marking_codes_out` (`codes`, `printed_codes[]` с
  `id/cis_code/has_label_artifact`, `quantity`, `shortage`, `layout`) — чтобы фронт
  переиспользовал существующую печать ленты.
- Доступ: как у `list_product_marking_codes` (`get_current_user` +
  `_assert_product_marking_access` через `get_effective_seller_id`).
- Блокировка строк обязательна: два оператора из одной корзины не должны получить одни КМ.
- Списание из общей корзины уменьшает доступность у всех товаров этой корзины — это
  корректно (код физически один).

### Часть B — фронтенд: гейт учитывает общие корзины

- `HonestSignScreen.tsx:683` → `markingAvailable: row.personal_available + Σ row.shared_baskets[].available`
  (данные уже есть в `ProductInventoryRow.shared_baskets`, поле `available`).
- `useFfProductMarkingPrint.tsx`: расширить тип `MarkingOverviewResponse` полем
  `shared_baskets: { available: number }[]` (бэкенд уже отдаёт `SharedBasketOverviewOut.available`);
  `markingAvailable = sumPersonalAvailable(personal_pools) + Σ shared_baskets[].available`.

### Часть C — фронтенд: каталожная печать становится списывающей

В `MarkingPrintDialog.tsx` `printCatalogTape`: заменить `GET .../products/{id}/codes` +
клиентскую выборку на `POST .../products/{id}/print` с
`{ quantity: canPrintCount, layout_json: layout, allow_partial: allowPartial }`; ответ
обработать как ветка `printLineTape` (`quantity < 1` → показать `shortage`/«нет доступных
КМ», иначе печать `data.codes` через существующий `deliverTape`); после успеха `ctx.onPrinted()`
(обновит доступно↓/напечатано↑). Гейт в модалке (`available`/`shortage`/`canPrintCount`)
уже работает от `ctx.markingAvailable` — станет корректным за счёт части B.

## 2C. Аудит: КМ ЧЗ должен списываться во ВСЕХ точках печати

Инвариант: печать НОВОГО КМ (не reprint) → атомарно `STATUS_PRINTED` + `EVENT_PRINTED`,
`available` уменьшается, `printed` растёт.

| Каталог товаров — `ProductBarcodePrintButton` → `openCatalogProductPrint` | `printCatalogTape` | **НЕТ** ✗ | Фикс частью C |

## Тесты (guard против регрессии)

Backend:
- инвариант списания на каждый живой путь: до печати запомнить `available`/`printed`;
  печать N КМ → `available -= N`, `printed += N`, у каждого КМ `status == printed` + событие
  `printed`;
- общая корзина (пул на 2 товара): печать для товара A списывает КМ; печать для товара B из
  той же корзины НЕ выдаёт уже напечатанные (берёт следующие `available`), дублей CIS нет;
- `allow_partial=false` при нехватке → `quantity: 0` + `shortage`, статусы КМ не изменились
  (откат);
- параллельная печать из одного пула не выдаёт один КМ дважды (`with_for_update`);
- reprint: `available` не меняется, событие `reprinted`.
Frontend:
- unit: `markingAvailable` = personal + Σ shared_baskets.available;
- e2e (рядом с `ff-honest-sign*.spec.ts`): импорт в общую корзину (2 товара на GTIN) →
  кнопка печати НЕ заблокирована, печать проходит, уходит списывающий POST, «Доступно»
  уменьшается после печати.

## Крайние случаи

- Нехватка: тот же контракт, что у упаковки (без `allow_partial` — блок с `shortage`;
  чекбокс «печатать доступные N» работает как сейчас).
- Разные seller: выборка строго по `product.seller_id`.
- GTIN не совпадает / пул не привязан к товару: доступно 0, печать блокируется штатно
  (отдельный кейс «коды в непривязанном пуле», не в scope).

## Критерии приёмки

1. Товар, чьи КМ только в общей корзине, печатается из «Честного знака» и из каталога без
   «не хватает N из N».
2. Любая печать нового КМ помечает его `printed` на бэкенде; повторная печать (того же или
   другого товара из той же корзины) не выдаёт уже напечатанные — дублей нет.
3. Нехватка обрабатывается тем же контрактом, что печать из упаковки.
4. Инвариант списания покрыт автотестом для каждого живого пути; deprecated-эндпоинты
   печати либо списывают, либо недоступны; мобилка проверена.
5. Путь печати из упаковки не изменён; существующие тесты зелёные (обновить лишь те, что
   завязаны на старое несписывающее поведение каталожной печати).
